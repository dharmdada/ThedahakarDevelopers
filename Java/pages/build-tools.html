Theory Deep Dive

Build tools as reproducibility engines

Mental Model

Maven and Gradle do more than download dependencies. They define the lifecycle of the project: compile, test, package, publish, run checks, and produce artifacts that CI and deployment systems can trust.

Dependency management is risk management. Each dependency adds functionality but also version conflicts, transitive libraries, security exposure, and upgrade responsibility. A good build file is explicit, minimal, and repeatable.

Wrappers matter because they make the build independent of whatever tool version happens to be installed on a developer's machine. The command should behave the same locally and in CI.

Principles That Matter

Maven vs Gradle

Maven
Convention
XML-based, strict lifecycle, predictable structure.
Gradle
Flexibility
DSL-based, faster incremental builds, custom tasks.
Both
Mature
Use central repositories, plugins, CI integration.

Pick one per repository and standardize. Mixed toolchains in one codebase usually increase maintenance overhead.

Minimal Maven pom.xml

XML
<project xmlns="http://maven.apache.org/POM/4.0.0">
  <modelVersion>4.0.0</modelVersion>
  <groupId>com.example</groupId>
  <artifactId>inventory-service</artifactId>
  <version>1.0.0</version>

  <properties>
    <maven.compiler.source>21</maven.compiler.source>
    <maven.compiler.target>21</maven.compiler.target>
    <junit.version>5.10.2</junit.version>
  </properties>

  <dependencies>
    <dependency>
      <groupId>org.junit.jupiter</groupId>
      <artifactId>junit-jupiter</artifactId>
      <version>${junit.version}</version>
      <scope>test</scope>
    </dependency>
  </dependencies>
</project>

Minimal Gradle build.gradle

Groovy
plugins {
    id 'java'
}

group = 'com.example'
version = '1.0.0'

java {
    toolchain {
        languageVersion = JavaLanguageVersion.of(21)
    }
}

repositories {
    mavenCentral()
}

dependencies {
    testImplementation 'org.junit.jupiter:junit-jupiter:5.10.2'
}

test {
    useJUnitPlatform()
}

Dependency Scopes and Configurations

  • Maven compile / Gradle implementation: needed by production code.
  • Maven provided / Gradle compileOnly: present at compile-time, not packaged.
  • Maven runtime / Gradle runtimeOnly: only needed at runtime.
  • Maven test / Gradle testImplementation: only for tests.
WARN
Avoid dependency leaks. Keep API contracts stable by exposing as little third-party type surface as possible from public methods.

Core Commands You Should Know

Shell
# Maven
mvn clean test
mvn clean package
mvn dependency:tree

# Gradle wrapper
./gradlew clean test
./gradlew build
./gradlew dependencies

Always use wrapper scripts in team projects (mvnw / gradlew) to lock tool versions in CI and developer machines.
