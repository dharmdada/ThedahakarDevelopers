Concurrency & Multithreading

Running multiple tasks simultaneously, safely sharing state, and modern async programming.

TH

Theory Deep Dive

Concurrency as coordinated progress

Mental Model

Concurrency means multiple tasks make progress during overlapping time periods. Parallelism means tasks literally execute at the same time. Java supports both, but correctness is harder than starting threads.

Shared mutable state is the core danger. Without coordination, one thread may see stale data, interleaved writes, or broken invariants. synchronized, locks, atomics, concurrent collections, and immutable data are different tools for controlling this danger.

Executors separate task submission from thread management. CompletableFuture separates composition from blocking. Virtual threads make blocking code scale better, but they do not remove the need to protect shared state.

Principles That Matter

  • Concurrency is about managing shared state. Synchronization is the primary tool to prevent race conditions.
  • Modern Java prefers higher-level abstractions like ExecutorService and CompletableFuture over raw Threads.
  • Deadlocks happen when two or more threads wait for each other to release locks forever.

Common Misunderstandings

  • Thinking 'volatile' makes operations atomic. It only guarantees visibility of changes across threads.
  • Calling Thread.run() instead of Thread.start(). (run() executes synchronously in the current thread).
  • Using outdated classes like Vector or Hashtable instead of ConcurrentHashMap or Collections.synchronizedList.

Mastery Checklist

  • You can explain the thread lifecycle states (New, Runnable, Blocked, Waiting, Terminated).
  • You know the difference between wait()/notify() and Thread.sleep().
  • You can implement a Thread-safe Singleton.

Threads vs Processes: The Kitchen Analogy

A Process is like a restaurant kitchen. It has its own dedicated space, ingredients (memory), and equipment. If one kitchen burns down, the other kitchens on the block are safe.

A Thread is like a chef working inside that kitchen. You can have multiple chefs (threads) chopping vegetables and cooking soup at the same time. Because they share the same kitchen (shared memory), they can work together very fast—but if they aren't careful, they might both grab the same knife at the same time and cause a disaster (a race condition).

JavaThread vs Runnable
// 1. Extend Thread class
class MyThread extends Thread {
    @Override
    public void run() { System.out.println("Thread running"); }
}
new MyThread().start(); // MUST call start(), not run()

// 2. Implement Runnable (Better — allows extending other classes)
class MyTask implements Runnable {
    @Override
    public void run() { System.out.println("Task running"); }
}
new Thread(new MyTask()).start();

// 3. With Lambda (Best for simple tasks)
Thread t = new Thread(() -> System.out.println("Lambda thread"));
t.start();

// Sleep and Join
Thread.sleep(1000); // Pause current thread for 1 second
t.join();           // Wait for thread t to finish

Executors (The Modern Way)

Managing threads manually is expensive and error-prone. Use thread pools.

JavaExecutorService
import java.util.concurrent.*;

// Create a pool of 4 threads
ExecutorService pool = Executors.newFixedThreadPool(4);

// Submit a Runnable (no return value)
pool.execute(() -> System.out.println("Running in pool"));

// Submit a Callable (returns a value via Future)
Future<Integer> future = pool.submit(() -> {
    Thread.sleep(1000);
    return 42;
});

// Do other work...
// Then get the result (blocks until ready)
try {
    int result = future.get(); // 42
} catch (InterruptedException | ExecutionException e) {
    e.printStackTrace();
}

// Always shutdown your pools!
pool.shutdown();

Synchronization: The Shared Bank Account

Imagine you and your spouse have a joint bank account with $100. You both go to different ATMs at the exact same millisecond and withdraw $100. If the bank software doesn't use Synchronization, both ATMs check the balance, see $100, and dispense the cash. The bank just lost $100. This is called a Race Condition.

Synchronization acts as a lock on the bank account. When you start your transaction, you lock the account. Your spouse's ATM must freeze and wait until you are completely finished before it is allowed to check the balance.

Java
class Counter {
    private int count = 0;

    // synchronized method: locks the whole object (this)
    public synchronized void increment() {
        count++;
    }

    public void incrementBlock() {
        // synchronized block: locks only specific code
        synchronized (this) {
            count++;
        }
    }
}

// Modern alternative: ReentrantLock
class LockedCounter {
    private int count = 0;
    private final Lock lock = new ReentrantLock();

    public void increment() {
        lock.lock();
        try {
            count++;
        } finally {
            lock.unlock(); // ALWAYS in finally block!
        }
    }
}

Atomic Variables & Concurrent Collections

For simple counters and collections, use built-in thread-safe classes instead of manual locks.

Java
// Atomic variables (lock-free thread-safe updates)
AtomicInteger atomicCount = new AtomicInteger(0);
atomicCount.incrementAndGet(); // Safe across threads
atomicCount.addAndGet(5);

// Concurrent Collections
// Fast, thread-safe map (better than Hashtable or synchronized map)
ConcurrentMap<String, Integer> map = new ConcurrentHashMap<>();
map.putIfAbsent("Key", 1);

// Thread-safe list (good for many reads, few writes)
List<String> safeList = new CopyOnWriteArrayList<>();

// Thread-safe queue
BlockingQueue<String> queue = new ArrayBlockingQueue<>(100);
queue.put("item");  // Blocks if full
queue.take();       // Blocks if empty

CompletableFuture: The Deli Ticket

In the old days, if you wanted the result of a background thread, you had to call future.get(), which would freeze (block) your current thread until the background job finished. This defeats the purpose of async programming!

CompletableFuture is like taking a ticket at a deli counter. You don't stand there staring at the butcher. You take your ticket, give the butcher a list of instructions ("When my sandwich is ready, add mayo, then wrap it, then text my phone"), and you walk away to do other shopping. The pipeline executes automatically when the data is ready.

JavaCompletableFuture
// Run async task
CompletableFuture<void> task1 = CompletableFuture.runAsync(() -> {
    System.out.println("Background task");
});

// Run async and return value
CompletableFuture<String> fetchUser = CompletableFuture.supplyAsync(() -> {
    return "Alice"; // Simulating DB call
});

// Chaining operations
fetchUser
    .thenApply(name -> name.toUpperCase())    // Transform
    .thenAccept(name -> System.out.println(name)) // Consume result
    .exceptionally(ex -> {                   // Error handling
        System.out.println("Error: " + ex);
        return null;
    });

// Combine two futures
CompletableFuture<Integer> price1 = CompletableFuture.supplyAsync(() -> 100);
CompletableFuture<Integer> price2 = CompletableFuture.supplyAsync(() -> 50);

price1.thenCombine(price2, (p1, p2) -> p1 + p2)
      .thenAccept(total -> System.out.println("Total: " + total));

Virtual Threads: The Infinite Waiters (Java 21)

Traditional Java threads are tied directly to OS threads. OS threads are incredibly heavy (taking ~1MB of memory each). If you try to spawn 100,000 OS threads, your computer will crash.

Virtual Threads (Project Loom) change everything. They are managed by the JVM, not the OS. They are so lightweight that you can easily spawn millions of them. You no longer need complex Thread Pools. The new rule in Java 21+ is: One thread per task.

JavaVirtual Threads
// Start a virtual thread
Thread vThread = Thread.startVirtualThread(() -> {
    System.out.println("Running in a virtual thread");
});

// Executor with virtual threads (Thread-per-task model)
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 10_000; i++) {
        executor.submit(() -> {
            Thread.sleep(Duration.ofSeconds(1));
            return 42;
        });
    }
} // Automatically waits for all 10,000 tasks to finish!

Real-World Use Case: High-Throughput Web Servers

In the past, Java web servers (like Tomcat) used a "Thread-per-request" model. If 1,000 users made a request at the same time, the server needed 1,000 OS threads. This consumed massive amounts of memory and caused the CPU to spend more time switching between threads (context switching) than actually doing work.

With Virtual Threads in Java 21, a single server can handle millions of concurrent requests. When a request waits for a database query to return, the virtual thread is "mounted" off the OS thread, allowing another virtual thread to run. This unlocks massive scalability without changing the familiar synchronous coding style!

Interview Preparation

Question 1
Runnable vs Callable?
Runnable's run() method returns void and cannot throw checked exceptions. Callable's call() method returns a Future result and can throw checked exceptions.
Question 2
What does the 'synchronized' keyword do?
It acquires an intrinsic lock on an object, ensuring that only one thread can execute a block of code on that object at a time.
