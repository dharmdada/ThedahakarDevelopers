Classes & Objects

The blueprint of everything in Java — fields, constructors, methods, access control.

TH

Theory Deep Dive

Classes as enforceable domain boundaries

Mental Model

A class bundles state and behavior under one name. Its job is to protect invariants: rules that must always remain true for objects of that type. Constructors establish invariants, methods preserve them, and visibility prevents invalid outside mutation.

Fields are implementation details unless there is a deliberate reason to expose them. Getters and setters are not automatic good design; they are only useful when reading or changing the value is part of the object's valid contract.

Static members belong to the class rather than an object. They are useful for constants and stateless helpers, but overuse of static state makes code harder to test and reason about.

The Blueprint Analogy

To understand Object-Oriented Programming (OOP), you must understand the difference between a Class and an Object. The easiest way to think about this is through the lens of architecture.

Imagine you are an architect designing a house. You draw a highly detailed blueprint that specifies how many doors the house has, what color the walls are, and where the plumbing goes. This blueprint is the Class. You cannot live inside a blueprint, and the blueprint itself doesn't take up any physical space on a street.

When a construction company takes your blueprint and builds an actual, physical house from it, that physical house is the Object (also known as an Instance). You can build thousands of distinct houses from the exact same blueprint. They might have different owners or different paint colors, but they all share the identical structure defined by the class.

In Java, every single piece of data and logic must reside inside a class. Let's look at the anatomical structure of a Java class.

JavaComplete Class Structure
public class Student {

    // ── FIELDS (State) ────────────────────────
    private String name;          // Instance field
    private int age;
    private static int count = 0; // Class field (shared)

    // ── CONSTRUCTOR ──────────────────────────
    public Student(String name, int age) {
        this.name = name;
        this.age = age;
        count++;
    }

    // ── METHODS (Behavior) ──────────────────
    public String getName() { return name; }
    public void setName(String name) { this.name = name; }

    // Static method — called on class, not instance
    public static int getCount() { return count; }

    // ── toString ────────────────────────────
    @Override
    public String toString() {
        return "Student{name='%s', age=%d}".formatted(name, age);
    }
}

// Creating objects
Student s1 = new Student("Alice", 20);
Student s2 = new Student("Bob", 22);
System.out.println(s1);           // Student{name='Alice', age=20}
System.out.println(Student.getCount()); // 2

Constructors: Bringing Objects to Life

When you use the new keyword (like new Student()), the JVM allocates memory on the Heap for the new object. But before you can use it, the object needs to be set up. This is the job of the Constructor.

A constructor is a special block of code that runs exactly once per object, at the exact moment the object is created. Its primary purpose is to initialize the object's state (its fields) so that the object starts its life in a valid, usable condition.

  • Naming Rule: Constructors must have the exact same name as the class.
  • No Return Type: Unlike methods, constructors do not have a return type (not even void).
  • Overloading: You can have multiple constructors in the same class, as long as they take different parameters.
JavaConstructor Types
public class Car {
    private String brand;
    private int year;

    // Default constructor (no args)
    public Car() {
        this("Unknown", 2024); // Constructor chaining with this()
    }

    // Parameterized constructor
    public Car(String brand, int year) {
        this.brand = brand;
        this.year = year;
    }

    // Copy constructor
    public Car(Car other) {
        this.brand = other.brand;
        this.year = other.year;
    }
}

Car c1 = new Car();                    // Default
Car c2 = new Car("Toyota", 2023);      // Parameterized
Car c3 = new Car(c2);                  // Copy
⚠️
No-Arg Constructor RuleIf you define NO constructors, Java provides a default no-arg one. But if you define ANY constructor, the default is NOT provided — you must explicitly add a no-arg constructor if you need one.

The this Keyword: Self-Reference

Imagine you are in a crowded room, and someone asks, "Who wants pizza?" If you want pizza, you don't say your own name ("John wants pizza!"); you say, "I want pizza."

In Java, the this keyword is how an object refers to itself. When code inside a method needs to access the fields or methods of the specific object that invoked it, it uses this.

The most common use case is resolving naming collisions. If a method parameter has the exact same name as a class field (which is very common in constructors and setters), the compiler gets confused. By prefixing the variable with this., you explicitly tell the compiler: "I am talking about the field that belongs to this object, not the parameter you just handed me."

Java
// 1. Distinguish field from parameter
public void setName(String name) {
    this.name = name; // this.name = field, name = parameter
}

// 2. Call another constructor (must be FIRST statement)
public Car() {
    this("Default", 2024); // Calls Car(String, int)
}

// 3. Return current object (for method chaining / fluent API)
public Car setBrand(String brand) {
    this.brand = brand;
    return this;
}
// Allows: new Car().setBrand("BMW").setYear(2024);

Static Members

static means "belongs to the class, not to any instance". Shared across all objects.

Java
public class MathUtils {
    // Static field — one copy shared by all instances
    public static final double PI = 3.14159265;

    // Static method — no need to create an object
    public static int max(int a, int b) {
        return (a > b) ? a : b;
    }

    // Static initializer block — runs once when class loads
    private static Map<String, Integer> cache;
    static {
        cache = new HashMap<>();
        cache.put("default", 0);
    }
}

// Called on the CLASS, not an object
int result = MathUtils.max(5, 10);
🔥
Static Cannot Access InstanceA static method cannot access instance fields or instance methods directly. It has no this reference. But an instance method CAN access static members.

Encapsulation: The First Pillar of OOP

Encapsulation is the practice of hiding the internal state (fields) of an object and requiring all interaction to be performed through an object's methods. But why is this necessary?

Imagine a BankAccount object. If the balance field is public, any other piece of code in your entire application can directly change it: account.balance = -999999;. This is incredibly dangerous. An object should be in complete control of its own destiny and its own data integrity.

To enforce this, we make the fields private (hiding them from the outside world), and expose public methods (Getters and Setters). This allows the class to inject validation logic whenever someone tries to change the data.

Java
public class BankAccount {
    private double balance; // Hidden from outside

    // Getter: read-only access
    public double getBalance() { return balance; }

    // Setter with VALIDATION — this is the power of encapsulation
    public void deposit(double amount) {
        if (amount <= 0) throw new IllegalArgumentException("Amount must be positive");
        this.balance += amount;
    }

    public void withdraw(double amount) {
        if (amount > balance) throw new IllegalArgumentException("Insufficient funds");
        this.balance -= amount;
    }
}

Inner & Nested Classes

Java
public class Outer {
    private int x = 10;

    // Inner class (non-static) — can access outer's fields
    class Inner {
        void show() { System.out.println(x); } // Accesses outer x!
    }

    // Static nested class — cannot access outer's instance fields
    static class StaticNested {
        void show() { System.out.println("I'm static nested"); }
    }

    void demo() {
        // Anonymous inner class
        Runnable r = new Runnable() {
            @Override
            public void run() { System.out.println("Anonymous!"); }
        };

        // Local class (inside a method)
        class LocalClass {
            void greet() { System.out.println("I'm local"); }
        }
    }
}

// Usage
Outer outer = new Outer();
Outer.Inner inner = outer.new Inner();        // Needs outer instance
Outer.StaticNested sn = new Outer.StaticNested(); // No outer needed

Final Classes & Methods

Java
// Final class: CANNOT be subclassed
public final class Constants {
    public static final int MAX = 100;
}
// class Sub extends Constants {}  →  ❌ Compile error!

// Final method: CANNOT be overridden by subclasses
class Base {
    public final void important() {
        System.out.println("Cannot change me");
    }
}

// Final variable: assigned once, cannot change
final int x = 10;
// x = 20;  →  ❌ Compile error!

Real-World Use Case: The Domain Model

In a standard Spring Boot Enterprise Application, Classes are heavily used to create the Domain Model. If you are building Twitter, you don't pass raw Strings and Integers around; you encapsulate everything into a User class, a Tweet class, and a FollowerGraph class.

By using Encapsulation, you ensure that someone cannot accidentally set a user's follower count to -500. The setFollowerCount() method will contain validation logic to prevent invalid states, protecting the entire architecture from bad data.

Interview Preparation

Question 1
What is the difference between a Class and an Object?
A Class is a blueprint (the architectural drawings) stored in Metaspace. An Object is an actual physical instance (the built house) stored in the Heap memory.
Question 2
Explain Encapsulation.
It is the bundling of data (fields) and methods that operate on that data into a single unit (class), while hiding the internal state by making fields private and requiring all interactions to happen via public methods (getters/setters).
Question 3
What does the 'static' keyword mean?
It means the field or method belongs to the Class itself, not to any specific instance. All objects share the exact same static field.
