Testing with JUnit 5 and Mockito

Write fast, reliable tests that document behavior, survive refactoring, and give you confidence to ship.

TH

Theory Deep Dive

Tests as executable specifications of behavior

Mental Model

Tests are not only defect detectors. They are executable specifications. A well-written test suite tells the next developer exactly what the code promises to do and enforces that promise during every refactoring.

The Testing Pyramid is a ratio rule:

  • Unit tests (70%+): Millisecond-fast, test one class, no I/O, no network. Cover all business logic branches.
  • Integration tests (20%): Verify that components wire together correctly — database queries, Spring context, API contracts.
  • End-to-end tests (10%): Full browser/HTTP flows. Expensive and slow. Cover critical user journeys only.

The most common failure mode is inverting the pyramid: writing integration tests for everything because they feel more "real". The result is a 20-minute CI pipeline that discourages running tests at all.

Mocks should be used at external boundaries only (HTTP clients, DB repositories, message queues). Over-mocking creates tests that mirror implementation — they pass when they shouldn't and fail when refactoring.

Principles That Matter

  • Arrange-Act-Assert (AAA) — Every test has exactly three phases. No interleaving setup and assertions.
  • Test behavior, not implementation — If renaming a private method breaks tests, the tests are testing internals.
  • One logical assertion per test — A test named register_shouldCreateUserAndSendEmail is testing two things. Split it.
  • Tests must be deterministic — No dependency on system clock, random, network, or test execution order.
  • Name tests as specifications: methodName_scenario_expectedResult

Common Misunderstandings

  • 100% coverage ≠ good tests — Testing getName() and setName() reaches coverage goals but finds no bugs.
  • Mocking the class under test — You are not testing the class if you mock it. Only mock its dependencies.
  • Tests that depend on order — JUnit 5 may run tests in any order. Each test must set up its own state completely.
  • Testing private methods — If a private method needs its own test, it belongs in a separate class. Test through the public API.

Mastery Checklist

  • You can write a JUnit 5 test with full lifecycle annotations.
  • You can use @ParameterizedTest with multiple data providers.
  • You can mock dependencies and verify interactions with Mockito.
  • You understand the difference between Mock, Stub, Spy, and Fake.
  • You can capture and assert on arguments passed to mocked methods.

JUnit 5 Full Lifecycle

JavaComplete test class structure
@DisplayName("OrderService — placing orders")
class OrderServiceTest {

    private OrderRepository   repo;
    private PaymentGateway    gateway;
    private OrderService      service;

    @BeforeAll
    static void initSuite() {
        // Runs ONCE before all tests in this class (static!)
        // Good for: starting test containers, loading shared fixtures
    }

    @BeforeEach
    void setUp() {
        // Runs before EACH test. Recreate mocks fresh every time.
        repo    = mock(OrderRepository.class);
        gateway = mock(PaymentGateway.class);
        service = new OrderService(repo, gateway);
    }

    @AfterEach
    void tearDown() {
        // Cleanup after each test (close resources, reset state)
    }

    @AfterAll
    static void cleanUpSuite() {
        // Stop test containers, close shared connections
    }

    @Test
    @DisplayName("placeOrder — should persist and charge when payment succeeds")
    void placeOrder_paymentSucceeds_shouldSaveAndCharge() {
        // Arrange
        Order order = new Order("ORD-001", 250.00);
        when(gateway.charge(order)).thenReturn(PaymentResult.success());

        // Act
        boolean result = service.placeOrder(order);

        // Assert
        assertTrue(result);
        verify(repo, times(1)).save(order);
        verify(gateway, times(1)).charge(order);
    }

    @Test
    @DisplayName("placeOrder — should NOT save when payment fails")
    void placeOrder_paymentFails_shouldNotSave() {
        Order order = new Order("ORD-002", 100.00);
        when(gateway.charge(order)).thenReturn(PaymentResult.failure("DECLINED"));

        boolean result = service.placeOrder(order);

        assertFalse(result);
        verify(repo, never()).save(any());
    }

    @Test
    @DisplayName("placeOrder — should throw when order amount is negative")
    void placeOrder_negativeAmount_shouldThrowIllegalArgument() {
        Order invalid = new Order("ORD-003", -1.00);

        assertThrows(IllegalArgumentException.class,
            () -> service.placeOrder(invalid));
    }
}

Parameterized Tests — Test Data Without Code Duplication

JavaMultiple data providers
class AgeValidatorTest {

    // Inline CSV data
    @ParameterizedTest(name = "age={0} → valid={1}")
    @CsvSource({
        "17, false",
        "18, true",
        "65, true",
        "130, false",
        "-1, false"
    })
    void isEligible_shouldValidateBoundaries(int age, boolean expected) {
        assertEquals(expected, AgeValidator.isEligible(age));
    }

    // External CSV file: src/test/resources/discount-cases.csv
    @ParameterizedTest
    @CsvFileSource(resources = "/discount-cases.csv", numLinesToSkip = 1)
    void applyDiscount_csvDriven(double base, double pct, double expected) {
        assertEquals(expected, PriceService.applyDiscount(base, pct), 0.001);
    }

    // Enum-driven: test all enum values
    @ParameterizedTest
    @EnumSource(value = UserRole.class, names = {"ADMIN", "SUPERUSER"})
    void hasAdminAccess_privilegedRoles_shouldReturnTrue(UserRole role) {
        assertTrue(AccessChecker.hasAdminAccess(role));
    }

    // Method source for complex objects
    static Stream<Arguments> provideOrders() {
        return Stream.of(
            Arguments.of(new Order("A", 0),    OrderStatus.INVALID),
            Arguments.of(new Order("B", 100),  OrderStatus.PENDING),
            Arguments.of(new Order("C", 10000), OrderStatus.REQUIRES_REVIEW)
        );
    }

    @ParameterizedTest
    @MethodSource("provideOrders")
    void classifyOrder_byAmount(Order order, OrderStatus expected) {
        assertEquals(expected, OrderClassifier.classify(order));
    }
}

Mockito: Mocking, Stubbing and Verification

JavaCore Mockito patterns
// --- Creating mocks ---
UserRepository repo = mock(UserRepository.class);
// Or with annotation: @Mock UserRepository repo; + @ExtendWith(MockitoExtension.class)

// --- Stubbing: define what mocks return ---
when(repo.findById(42L)).thenReturn(Optional.of(new User(42L, "Alice")));
when(repo.findById(99L)).thenReturn(Optional.empty());
when(repo.save(any(User.class))).thenReturn(savedUser);

// Throw on call
when(repo.findById(-1L)).thenThrow(new IllegalArgumentException("Invalid ID"));

// Return different values on successive calls
when(repo.generateId())
    .thenReturn(1L)
    .thenReturn(2L)
    .thenReturn(3L);

// --- Verification: did the mock get called correctly? ---
verify(repo, times(1)).save(any());
verify(repo, never()).delete(any());
verify(repo, atLeast(2)).findById(anyLong());
verify(repo, atMostOnce()).findAll();

// Verify order of calls
InOrder inOrder = inOrder(repo, emailService);
inOrder.verify(repo).save(any());
inOrder.verify(emailService).sendWelcome(any());

// --- ArgumentCaptor: inspect what was passed ---
ArgumentCaptor<User> captor = ArgumentCaptor.forClass(User.class);
verify(repo).save(captor.capture());
User savedUser = captor.getValue();
assertEquals("alice@example.com", savedUser.email());

Test Doubles: Mock vs Stub vs Spy vs Fake

Mock
Verify calls
Created with Mockito. Returns null by default. You verify it was called with the right args.
Stub
Return canned data
Programmed to return fixed values. No behavior verification. Used for setup only.
Spy
Partial real behavior
Wraps a real object. Calls real methods unless stubbed. Use sparingly.
Fake
Lightweight impl
A working but simplified implementation (e.g., in-memory HashMap as a repository). Best for integration-level tests.
JavaSpy example: wrapping real object, overriding one method
List<String> realList = new ArrayList<>();
List<String> spyList  = spy(realList);

// Real behavior unless stubbed
spyList.add("hello");
assertEquals(1, spyList.size()); // calls real size()

// Override specific behavior
doReturn(100).when(spyList).size(); // dangerous! use sparingly
assertEquals(100, spyList.size());  // stubbed
WARN
Prefer Mock over Spy. Spies call real code by default, which can trigger unexpected side effects and slow down tests. Use them only when you genuinely need partial real behavior.

Controlling Time in Tests

Never call LocalDateTime.now() or Instant.now() directly in production code you want to test. Inject a Clock:

JavaTestable clock injection pattern
// Production code takes a Clock
class SubscriptionService {
    private final Clock clock;
    SubscriptionService(Clock clock) { this.clock = clock; }

    boolean isExpired(Subscription sub) {
        return sub.expiresAt().isBefore(LocalDate.now(clock));
    }
}

// Test: inject a fixed clock
class SubscriptionServiceTest {
    @Test
    void isExpired_pastExpiryDate_shouldReturnTrue() {
        Clock fixedClock = Clock.fixed(
            Instant.parse("2026-06-01T00:00:00Z"),
            ZoneOffset.UTC
        );
        SubscriptionService svc = new SubscriptionService(fixedClock);
        Subscription expired = new Subscription(LocalDate.of(2026, 1, 1));

        assertTrue(svc.isExpired(expired));
    }
}

// Production wiring
new SubscriptionService(Clock.systemUTC());

CI Test Quality Checklist

  • Tests pass with mvn test or gradle test in a clean environment with no manual setup.
  • No test depends on another test's side effects — full isolation.
  • System clock, random seeds, and external calls are all controlled/mocked.
  • Assertions produce useful messages: assertEquals(expected, actual, "Reason for failure").
  • Flaky tests are tracked and fixed immediately — a flaky test is worse than no test.
  • Test execution time for unit suite stays under 30 seconds.

Interview Preparation

Question 1
What is a Mock and when do you use it?
A fake object that records calls and returns configured responses. Use mocks to isolate the class under test from its external dependencies (DB, HTTP, message queues). Do NOT mock things the class owns internally.
Question 2
Unit vs Integration vs E2E Testing?
Unit: one class, no I/O, milliseconds. Integration: multiple components + real DB/infra. E2E: full user journey through UI/HTTP. Follow the pyramid: mostly unit, some integration, few E2E.
Question 3
How do you test time-dependent code?
Inject a java.time.Clock dependency instead of calling Instant.now() directly. In tests, inject Clock.fixed(...) to control time precisely. In production, inject Clock.systemUTC().
Question 4
What is ArgumentCaptor?
A Mockito tool that captures the actual argument passed to a mocked method so you can run detailed assertions on it. Useful when the argument is built internally and not accessible from the test directly.
