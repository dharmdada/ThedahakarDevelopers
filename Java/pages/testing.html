Theory Deep Dive

Testing as executable design feedback

Mental Model

Tests are not only defect detectors. They are executable statements of expected behavior. A well-written test explains what a class promises to do and protects that promise while the code evolves.

Unit tests should isolate logic and run fast. Integration tests should verify wiring and external boundaries. End-to-end tests should cover critical journeys. Mixing these purposes creates slow, fragile suites that developers stop trusting.

Mocks are useful at boundaries but dangerous when they mirror implementation too closely. If a test knows every internal call, refactoring becomes painful even when behavior stays correct.

Principles That Matter

  • Understand the purpose of the feature before memorizing its syntax.
  • Connect the concept to runtime behavior, API design, and failure cases.
  • Prefer readable code that makes assumptions explicit.
  • Use the feature when it reduces complexity, not just because it exists.

Common Misunderstandings

  • Learning only the code pattern without the reason behind it.
  • Ignoring edge cases until they appear as bugs.
  • Using the feature everywhere instead of where it fits the problem.
  • Skipping the connection between this topic and surrounding Java concepts.

Mastery Checklist

  • You can explain testing in plain language without reading code.
  • You can identify the right and wrong use cases for the concept.
  • You can debug common failures related to the topic.
  • You can teach the concept using a small real-world example.

Test Strategy: Pyramid First

  • Unit tests: most of your suite, very fast, isolate business logic.
  • Integration tests: fewer, validate database and framework wiring.
  • End-to-end tests: minimal, verify key user paths only.

A common failure mode is overusing slow integration tests and under-testing pure logic. Push business rules into unit-testable services.

JUnit 5 Structure

JavaService Unit Test
import static org.junit.jupiter.api.Assertions.*;
import org.junit.jupiter.api.Test;

class PriceServiceTest {
    @Test
    void applyDiscount_shouldRoundToTwoDecimals() {
        PriceService service = new PriceService();
        double result = service.applyDiscount(19.99, 10);

        assertEquals(17.99, result, 0.001);
    }
}
TIP
Test names should describe behavior, not implementation: calculateTax_whenStateIsCA_appliesStateRate.

Parameterized Tests for Edge Cases

Java
import static org.junit.jupiter.api.Assertions.*;
import org.junit.jupiter.params.ParameterizedTest;
import org.junit.jupiter.params.provider.CsvSource;

class AgeValidatorTest {
    @ParameterizedTest
    @CsvSource({
        "17,false",
        "18,true",
        "65,true",
        "130,false"
    })
    void isAllowedAge_shouldValidateRange(int age, boolean expected) {
        assertEquals(expected, AgeValidator.isAllowed(age));
    }
}

Mocking External Dependencies with Mockito

Java
import static org.mockito.Mockito.*;
import static org.junit.jupiter.api.Assertions.*;
import org.junit.jupiter.api.Test;

class OrderServiceTest {
    @Test
    void placeOrder_shouldPersistOrderWhenPaymentSucceeds() {
        PaymentGateway gateway = mock(PaymentGateway.class);
        OrderRepository repo = mock(OrderRepository.class);
        when(gateway.charge(any())).thenReturn(true);

        OrderService service = new OrderService(gateway, repo);
        boolean created = service.placeOrder(new Order("A-101"));

        assertTrue(created);
        verify(repo).save(any(Order.class));
        verify(gateway).charge(any());
    }
}
WARN
Do not over-mock internal logic. Mock only boundaries (HTTP clients, DB repositories, queues). Pure business classes should be tested directly.

CI Test Quality Checklist

  • Tests run with a single command in clean environment.
  • No order dependency between tests.
  • Clock, randomness, and external network calls are controlled.
  • Assertions verify behavior, not exact logging text unless required.
  • Failing tests produce clear messages for quick diagnosis.
