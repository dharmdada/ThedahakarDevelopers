Design Patterns and SOLID

Battle-tested solutions to recurring design problems. Learn when to apply them — and crucially, when not to.

TH

Theory Deep Dive

Patterns as names for recurring tradeoffs

Mental Model

Design patterns are reusable names for common design shapes. Their value is communication: when you say "we used the Strategy pattern here", every experienced Java developer immediately understands the structure — interchangeable algorithms behind a common interface, injected at construction time.

The GoF (Gang of Four) book catalogued 23 patterns in 3 groups: Creational (how objects are built), Structural (how objects are composed), and Behavioral (how objects communicate).

Patterns should emerge from pressure in the code, not be applied upfront. The most common mistake is applying patterns before the problem exists. The second most common mistake is not recognizing when a pattern would save you from a growing mess.

In modern Java, many classic patterns have cleaner implementations. Strategy is just a functional interface + lambda. Command is a Runnable or Supplier. Observer is an event bus or reactive stream. Know the pattern conceptually, apply it with modern Java idioms.

Beginner-Friendly, Expert-Useful Lens

If you are new: Beginners should learn the problem each pattern solves before memorizing class diagrams.

If you are revising: Experts should refresh when patterns emerge naturally, when they overfit, and how modern Java changes older implementations.

Mastery check: You understand Design Patterns when you can explain the simple mental model, write a tiny runnable example, identify the common trap, and describe how it behaves in a real project.

Principles That Matter

  • Patterns are tools, not goals. Simple code that solves the problem is always better than elaborate pattern-driven code that doesn't need to be.
  • Favor composition over inheritance. Strategy, Decorator, and most behavioral patterns express this principle.
  • Code to interfaces. All patterns work best when caller and implementation are decoupled by an interface or abstract type.
  • SOLID gives you diagnostic tools, not dogma. Apply them when change pain reveals the problem.

Common Misunderstandings

  • Singleton misuse — Singletons that hold mutable state are global variables. They destroy testability and hide dependencies. Use dependency injection instead.
  • Factory vs Abstract Factory — Factory creates one type of product. Abstract Factory creates families of related products (e.g., UI widgets for Windows vs Mac).
  • Pattern obsession — Wrapping a simple 3-line function in Strategy + Factory + Builder is over-engineering. Don't do it.
  • Deep inheritance chains — "Diamond of death" and fragile base class problems. Composition breaks the chain.

Mastery Checklist

  • You can implement a thread-safe Singleton (double-checked locking or enum).
  • You can implement Builder for a class with 5+ optional fields.
  • You can explain Strategy and how Spring's dependency injection is built on it.
  • You can describe when each of the 6 key patterns applies.
  • You can name all 5 SOLID principles and give a real-world violation example for each.

Advanced Notes Most Courses Skip

Patterns are names for tradeoffs, not badges to attach to every problem.

Expert refresh: Experts should refresh when patterns emerge naturally, when they overfit, and how modern Java changes older implementations.

Hard-Won Tips

  • A pattern is useful only when it reduces change cost. If it adds indirection without a likely variation point, it is ceremony.
  • Strategy is often just an interface plus a few implementations; do not overbuild it.
  • Singleton is global state with a formal outfit. It should be rare, carefully initialized, and easy to test around.
  • Factory methods are valuable when construction choices are complex or need names; otherwise constructors and records are clearer.
  • Modern Java features can simplify classic patterns: records for value objects, lambdas for strategies, sealed types for known variants.

SOLID Principles with Real Java Examples

S — Single Responsibility Principle

A class should have one reason to change. If it handles both business logic and email sending, it changes when business rules change AND when email templates change.

Java❌ Violation — UserService does everything
record User(String email) {}

class UserService {
    void register(User user) {
        if (user.email() == null || !user.email().contains("@")) {
            throw new IllegalArgumentException("Invalid email");
        }
        System.out.println("Saving " + user.email());
        System.out.println("Sending welcome email");
        System.out.println("Writing audit log");
    }
}

public class Main {
    public static void main(String[] args) {
        new UserService().register(new User("alice@example.com"));
    }
}
Java✅ Fixed — each class has one job
record User(String email) {}
interface UserRepository { void save(User user); }
interface EmailService { void sendWelcome(User user); }
interface AuditLogger { void log(String event); }

class UserRegistrationService {
    private final UserRepository repo;
    private final EmailService email;
    private final AuditLogger audit;

    UserRegistrationService(UserRepository repo, EmailService email, AuditLogger audit) {
        this.repo = repo;
        this.email = email;
        this.audit = audit;
    }

    void register(User user) {
        repo.save(user);
        email.sendWelcome(user);
        audit.log("registered " + user.email());
    }
}

public class Main {
    public static void main(String[] args) {
        var service = new UserRegistrationService(
            user -> System.out.println("Saved " + user.email()),
            user -> System.out.println("Welcome " + user.email()),
            System.out::println
        );
        service.register(new User("alice@example.com"));
    }
}

O — Open/Closed Principle

Classes should be open for extension, closed for modification. Add behavior via new classes, not by editing existing ones.

Java✅ Add a new discount type without touching DiscountCalculator
interface DiscountPolicy {
    double apply(double price);
}

class NoDiscount implements DiscountPolicy {
    public double apply(double price) { return price; }
}

class BlackFridayDiscount implements DiscountPolicy {
    public double apply(double price) { return price * 0.70; }
}

class CheckoutService {
    private final DiscountPolicy policy;
    CheckoutService(DiscountPolicy policy) { this.policy = policy; }
    double total(double price) { return policy.apply(price); }
}

public class Main {
    public static void main(String[] args) {
        CheckoutService checkout = new CheckoutService(new BlackFridayDiscount());
        System.out.println(checkout.total(100.0));
    }
}

L — Liskov Substitution Principle

Subclasses must be fully substitutable for their base type without breaking caller behavior.

Java❌ Classic LSP violation: Square extends Rectangle
class Rectangle {
    protected int width;
    protected int height;
    void setWidth(int width) { this.width = width; }
    void setHeight(int height) { this.height = height; }
    int area() { return width * height; }
}

class Square extends Rectangle {
    @Override void setWidth(int width) { this.width = width; this.height = width; }
    @Override void setHeight(int height) { this.width = height; this.height = height; }
}

public class Main {
    static void resize(Rectangle rectangle) {
        rectangle.setWidth(5);
        rectangle.setHeight(4);
        System.out.println("Area: " + rectangle.area());
    }

    public static void main(String[] args) {
        resize(new Rectangle());
        resize(new Square());
    }
}

I — Interface Segregation Principle

No class should be forced to implement methods it doesn't use. Split fat interfaces.

Java✅ Split into focused interfaces
interface Printer { void print(); }
interface ScannerDevice { void scan(); }
interface FaxMachine { void fax(); }

class SimplePrinter implements Printer {
    public void print() { System.out.println("Printing"); }
}

class OfficeMachine implements Printer, ScannerDevice, FaxMachine {
    public void print() { System.out.println("Printing"); }
    public void scan() { System.out.println("Scanning"); }
    public void fax() { System.out.println("Faxing"); }
}

public class Main {
    public static void main(String[] args) {
        Printer printer = new SimplePrinter();
        printer.print();
    }
}

D — Dependency Inversion Principle

Depend on abstractions (interfaces), not concrete classes. High-level modules should not depend on low-level modules.

Java✅ OrderService depends on Repository interface, not MySQL impl
import java.util.*;

record Order(long id) {}
interface OrderRepository {
    void save(Order order);
    Optional<Order> findById(long id);
}

class InMemoryOrderRepository implements OrderRepository {
    private final Map<Long, Order> orders = new HashMap<>();
    public void save(Order order) { orders.put(order.id(), order); }
    public Optional<Order> findById(long id) { return Optional.ofNullable(orders.get(id)); }
}

class OrderService {
    private final OrderRepository repo;
    OrderService(OrderRepository repo) { this.repo = repo; }
    void create(Order order) { repo.save(order); }
}

public class Main {
    public static void main(String[] args) {
        OrderRepository repo = new InMemoryOrderRepository();
        OrderService service = new OrderService(repo);
        service.create(new Order(101));
        System.out.println(repo.findById(101));
    }
}

Creational Patterns

Singleton — Thread-Safe Implementation

JavaBest option: Enum Singleton (serialization + reflection safe)
enum AppConfig {
    INSTANCE;

    private final String dbUrl = System.getenv().getOrDefault("DB_URL", "jdbc:h2:mem:test");

    String dbUrl() { return dbUrl; }
}

public class Main {
    public static void main(String[] args) {
        System.out.println(AppConfig.INSTANCE.dbUrl());
    }
}
JavaAlternative: Initialization-on-demand holder (lazy, thread-safe)
class ConnectionPool {
    private ConnectionPool() {}

    private static class Holder {
        private static final ConnectionPool INSTANCE = new ConnectionPool();
    }

    static ConnectionPool getInstance() {
        return Holder.INSTANCE;
    }
}

public class Main {
    public static void main(String[] args) {
        System.out.println(ConnectionPool.getInstance() == ConnectionPool.getInstance());
    }
}

Builder — Complex Object Construction

JavaBuilder pattern — eliminates telescoping constructors
class EmailMessage {
    private final String to;
    private final String subject;
    private final String body;
    private final String replyTo;
    private final boolean html;

    private EmailMessage(Builder builder) {
        this.to = builder.to;
        this.subject = builder.subject;
        this.body = builder.body;
        this.replyTo = builder.replyTo;
        this.html = builder.html;
    }

    static class Builder {
        private final String to;
        private String subject = "";
        private String body = "";
        private String replyTo = "";
        private boolean html;

        Builder(String to) { this.to = to; }
        Builder subject(String subject) { this.subject = subject; return this; }
        Builder body(String body) { this.body = body; return this; }
        Builder replyTo(String replyTo) { this.replyTo = replyTo; return this; }
        Builder html(boolean html) { this.html = html; return this; }
        EmailMessage build() { return new EmailMessage(this); }
    }

    @Override public String toString() { return subject + " -> " + to + " html=" + html; }
}

public class Main {
    public static void main(String[] args) {
        EmailMessage message = new EmailMessage.Builder("alice@example.com")
            .subject("Welcome")
            .body("Hello Alice")
            .html(false)
            .build();
        System.out.println(message);
    }
}

Factory Method — Object Creation by Type

JavaFactory with modern switch expression
interface Notification {
    void send(String message);
}

class EmailNotification implements Notification {
    public void send(String message) { System.out.println("Email: " + message); }
}

abstract class NotificationDecorator implements Notification {
    protected final Notification wrapped;
    NotificationDecorator(Notification wrapped) { this.wrapped = wrapped; }
}

class LoggingNotification extends NotificationDecorator {
    LoggingNotification(Notification wrapped) { super(wrapped); }
    public void send(String message) {
        System.out.println("Log before send");
        wrapped.send(message);
    }
}

public class Main {
    public static void main(String[] args) {
        Notification notification = new LoggingNotification(new EmailNotification());
        notification.send("Build complete");
    }
}

Structural & Behavioral Patterns

Strategy — Algorithm Selection at Runtime

JavaStrategy with Java functional interface (modern, clean)
import java.util.*;

@FunctionalInterface
interface SortStrategy {
    void sort(List<Integer> data);
}

class NumberSorter {
    private final SortStrategy strategy;
    NumberSorter(SortStrategy strategy) { this.strategy = strategy; }
    void sort(List<Integer> data) { strategy.sort(data); }
}

public class Main {
    public static void main(String[] args) {
        List<Integer> numbers = new ArrayList<>(List.of(3, 1, 2));
        NumberSorter sorter = new NumberSorter(Collections::sort);
        sorter.sort(numbers);
        System.out.println(numbers);
    }
}

Decorator — Add Behavior Without Subclassing

JavaDecorator: wrapping with added behavior
interface TextProcessor {
    String process(String text);
}

class TrimProcessor implements TextProcessor {
    public String process(String text) { return text.trim(); }
}

class UppercaseProcessor implements TextProcessor {
    public String process(String text) { return text.toUpperCase(); }
}

class ProcessorChain implements TextProcessor {
    private final TextProcessor first;
    private final TextProcessor second;
    ProcessorChain(TextProcessor first, TextProcessor second) {
        this.first = first;
        this.second = second;
    }
    public String process(String text) { return second.process(first.process(text)); }
}

public class Main {
    public static void main(String[] args) {
        TextProcessor processor = new ProcessorChain(new TrimProcessor(), new UppercaseProcessor());
        System.out.println(processor.process("  hello  "));
    }
}

Observer — Decoupled Event Notification

JavaObserver with functional listeners (modern Java style)
import java.util.*;
import java.util.concurrent.CopyOnWriteArrayList;
import java.util.function.Consumer;

record Order(long id) {}

class OrderEventBus {
    private final List<Consumer<Order>> listeners = new CopyOnWriteArrayList<>();

    public void subscribe(Consumer<Order> listener) { listeners.add(listener); }
    public void publish(Order order) { listeners.forEach(listener -> listener.accept(order)); }
}

public class Main {
    public static void main(String[] args) {
        OrderEventBus bus = new OrderEventBus();
        bus.subscribe(order -> System.out.println("Audit order " + order.id()));
        bus.subscribe(order -> System.out.println("Email for order " + order.id()));
        bus.publish(new Order(42));
    }
}

Adapter — Wrap Incompatible Interfaces

JavaAdapting legacy payment gateway to modern interface
record Money(String currency, long cents) {}
record PaymentResult(boolean success, String message) {}

interface PaymentGateway {
    PaymentResult charge(String customerId, Money amount);
}

class ThirdPartyPaymentClient {
    String makePayment(String customerId, long cents, String currency) {
        return "OK:" + customerId + ":" + cents + currency;
    }
}

class ThirdPartyPaymentAdapter implements PaymentGateway {
    private final ThirdPartyPaymentClient client;
    ThirdPartyPaymentAdapter(ThirdPartyPaymentClient client) { this.client = client; }

    public PaymentResult charge(String customerId, Money amount) {
        String response = client.makePayment(customerId, amount.cents(), amount.currency());
        return new PaymentResult(response.startsWith("OK"), response);
    }
}

public class Main {
    public static void main(String[] args) {
        PaymentGateway gateway = new ThirdPartyPaymentAdapter(new ThirdPartyPaymentClient());
        System.out.println(gateway.charge("cust-1", new Money("USD", 2500)));
    }
}

Anti-Patterns to Actively Avoid

God Class
1000+ lines
One class knows everything and does everything. Change becomes impossible to isolate.
Anemic Domain
No behavior
Domain objects are just data bags (only getters/setters). All logic leaks into services.
Service Locator
Hidden deps
Calling a global registry for dependencies hides coupling. Prefer constructor injection.
Primitive Obsession
Stringly-typed
Using String for everything (email, phone, status) instead of value types. Loses validation.
INFO
Rule of thumb: start simple. Extract a pattern only when: (1) behavior varies across callers, (2) construction is getting complex, or (3) coupling makes testing painful.

Interview Preparation

Question 1
How do you implement a thread-safe Singleton?
Best: use enum — JVM guarantees single instance, handles serialization and reflection attacks. Alternative: initialization-on-demand holder (static inner class). Double-checked locking is acceptable but verbose.
Question 2
Strategy Pattern vs Inheritance?
Strategy composes behavior via an injected interface — swappable at runtime, fully testable with mocks. Inheritance bakes behavior in at compile time — changing algorithms requires new subclasses and breaks LSP when overdone.
Question 3
What is Dependency Injection?
A class receives its dependencies from the outside (via constructor or setter) rather than creating them internally. This decouples creation from use, makes testing easy (inject mocks), and respects Dependency Inversion. Spring is a DI framework.
Question 4
Decorator vs Inheritance?
Decorator adds behavior at runtime by wrapping an object. Inheritance is static — set at compile time. Decorator avoids subclass explosion when you have multiple independent optional behaviors (e.g., logging + caching + retry = 3 decorators vs 8 subclasses).
