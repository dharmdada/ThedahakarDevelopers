Design Patterns and SOLID

How to build Java systems that stay readable and changeable as complexity grows.

TH

Theory Deep Dive

Patterns as names for recurring tradeoffs

Mental Model

Design patterns are reusable names for common design shapes. They are useful because they let developers discuss structure quickly: strategy means interchangeable algorithms, adapter means incompatible interface wrapper, facade means simplified subsystem entry point.

Patterns should emerge from pressure in the code. If behavior varies, strategy may help. If construction is complex, builder may help. If a subsystem is noisy, facade may help. Applying patterns before the problem appears creates unnecessary ceremony.

SOLID principles are not rules to obey mechanically. They are diagnostic tools for change pain. When small changes require edits everywhere, responsibilities and dependencies are probably wrong.

Principles That Matter

  • Understand the purpose of the feature before memorizing its syntax.
  • Connect the concept to runtime behavior, API design, and failure cases.
  • Prefer readable code that makes assumptions explicit.
  • Use the feature when it reduces complexity, not just because it exists.

Common Misunderstandings

  • Learning only the code pattern without the reason behind it.
  • Ignoring edge cases until they appear as bugs.
  • Using the feature everywhere instead of where it fits the problem.
  • Skipping the connection between this topic and surrounding Java concepts.

Mastery Checklist

  • You can explain design patterns in plain language without reading code.
  • You can identify the right and wrong use cases for the concept.
  • You can debug common failures related to the topic.
  • You can teach the concept using a small real-world example.

SOLID Principles (Practical Lens)

  • S - Single Responsibility: one reason to change per class.
  • O - Open/Closed: add behavior via extension, not repeated edits.
  • L - Liskov Substitution: subclasses must preserve base contract.
  • I - Interface Segregation: many focused interfaces beat one giant interface.
  • D - Dependency Inversion: depend on abstractions, not concrete implementations.

SOLID is not about creating many classes blindly. It is about controlling change impact and reducing regression risk.

Structural and Behavioral Patterns

  • Adapter: wrap incompatible interfaces.
  • Decorator: add behavior without subclass explosion.
  • Facade: simplify a complex subsystem behind one API.
  • Strategy: choose algorithm at runtime.
  • Observer: publish/subscribe event flow.
  • Command: represent operations as objects (queue, retry, undo).

Strategy Pattern Example

Java
interface PricingStrategy {
    double calculate(double basePrice);
}

class RegularPricing implements PricingStrategy {
    public double calculate(double basePrice) { return basePrice; }
}

class PremiumPricing implements PricingStrategy {
    public double calculate(double basePrice) { return basePrice * 0.9; }
}

class PriceCalculator {
    private final PricingStrategy strategy;
    PriceCalculator(PricingStrategy strategy) { this.strategy = strategy; }
    double finalPrice(double basePrice) { return strategy.calculate(basePrice); }
}

Same calculator, different policy. This is easier to test and extend than a giant if/else chain.

Common Anti-Patterns to Avoid

  • God classes: too many responsibilities in one class.
  • Deep inheritance chains: fragile and hard to reason about.
  • Premature abstraction: interfaces with one implementation and no actual variation.
  • Pattern obsession: choosing a pattern before understanding the problem.
INFO
Rule of thumb: start simple, then extract pattern when duplication and change pressure become clear.
