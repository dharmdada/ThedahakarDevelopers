Setup and Hello World

Install Java correctly, understand the compile-run cycle, and build a toolchain that works across IDEs, terminals, and CI pipelines.

TH

Theory Deep Dive

Understanding the JVM ecosystem from source to execution

Mental Model: Compile Once, Run Anywhere

Java's core value proposition is platform independence. Your source code (.java) is compiled to bytecode (.class), not to native machine code. The bytecode is executed by the Java Virtual Machine (JVM) on any operating system that has a JVM installed.

This is the pipeline:

Pipeline
HelloWorld.java
  → javac (Java Compiler)
  → HelloWorld.class (JVM bytecode)
  → java (JVM — interprets + JIT-compiles bytecode to native)
  → Output on your screen

The JVM also provides: garbage collection, security sandboxing, JIT (Just-In-Time) compilation for hot code paths, and dynamic class loading. The JVM is itself a complex, optimized runtime — not a simple interpreter.

Beginner-Friendly, Expert-Useful Lens

If you are new: Beginners should learn the difference between installing Java, compiling code, running code, and configuring an IDE.

If you are revising: Experts should verify toolchains the way CI does: exact JDK version, explicit JAVA_HOME, deterministic builds, and clean environment assumptions.

Mastery check: You understand Setup and Toolchain when you can explain the simple mental model, write a tiny runnable example, identify the common trap, and describe how it behaves in a real project.

JDK vs JRE vs JVM

JVM
Runtime Engine
Executes bytecode. Includes garbage collector, JIT compiler, security manager. Not standalone — part of JRE/JDK.
JRE
Java Runtime
JVM + standard libraries. Enough to RUN Java programs. Deprecated since Java 11 — no longer distributed separately.
JDK
Developer Kit
JRE + javac + javadoc + jlink + profiling tools. Everything you need to BUILD and RUN Java. This is what you install.
OpenJDK
Open Source
The reference JDK implementation. Oracle JDK, Adoptium (Eclipse Temurin), Amazon Corretto, Azul Zulu are all built from OpenJDK.
INFO
Since Java 11, there is no separate JRE download. The JDK is all you install. On servers, you can create minimal custom runtimes using jlink.

Which Java Version to Use?

  • Java 21 (LTS) — Current LTS. Includes Virtual Threads (Project Loom), Pattern Matching, Records, Sealed Classes. Use this for new projects.
  • Java 17 (LTS) — Previous LTS. Widely supported by all frameworks. Still a solid choice.
  • Java 11 (LTS) — The baseline for many enterprise environments. HttpClient, var, local variable inference.
  • Avoid Java 8 for new code — it is in extended commercial support only. Major language features (Records, Sealed, Pattern Matching) require 17+.

Principles That Matter

  • Set JAVA_HOME explicitly. Tools like Maven, Gradle, and Tomcat rely on it.
  • Use build tool wrappers (mvnw, gradlew) — they download the correct Maven/Gradle version automatically and make builds reproducible.
  • Pin your Java version in the build file — sourceCompatibility = 21 in Gradle, <java.version>21</java.version> in Maven.

Common Misunderstandings

  • "I need to download a JRE" — No. Since Java 11, just install the JDK. It includes everything.
  • Multiple JDKs installed with no management — The one in PATH first wins. Use SDKMAN (Linux/Mac) or manually manage JAVA_HOME on Windows.
  • IDE compiled, terminal didn't — The IDE may be using a different JDK than your terminal. Always verify with java -version in the same terminal you run builds.

Mastery Checklist

  • You can verify JDK installation and version from the command line.
  • You can compile and run a Java file manually without an IDE.
  • You understand what JAVA_HOME is and why it matters.
  • You can configure IntelliJ IDEA or VS Code to use a specific JDK.
  • You can create a Maven or Gradle project from scratch.

Installing a JDK

ShellWindows — Winget (recommended)
# Install Java 21 (Eclipse Temurin — free, open source, production-grade)
winget install EclipseAdoptium.Temurin.21.JDK

# Or Oracle JDK 21
winget install Oracle.JDK.21

# Verify
java -version
javac -version
ShellLinux/macOS — SDKMAN (manage multiple versions)
# Install SDKMAN
curl -s "https://get.sdkman.io" | bash
source "$HOME/.sdkman/bin/sdkman-init.sh"

# List available Java versions
sdk list java

# Install Java 21 (Temurin)
sdk install java 21.0.3-tem

# Switch between versions
sdk use java 17.0.11-tem

# Set default
sdk default java 21.0.3-tem

# Current version
sdk current java

Environment Variables

ShellWindows — Set JAVA_HOME permanently
# Open System Properties → Advanced → Environment Variables
# New System Variable:
#   JAVA_HOME = C:\Program Files\Eclipse Adoptium\jdk-21.0.3.9-hotspot

# Edit PATH variable — Add at the top:
#   %JAVA_HOME%\bin

# Verify in a NEW terminal window:
echo %JAVA_HOME%
java -version
javac -version
ShellLinux/macOS — ~/.bashrc or ~/.zshrc
export JAVA_HOME="$HOME/.sdkman/candidates/java/current"
export PATH="$JAVA_HOME/bin:$PATH"

# Reload
source ~/.bashrc  # or source ~/.zshrc

# Verify
echo $JAVA_HOME
java -version
WARN
Most setup bugs come from PATH conflicts. If you have multiple JDK versions installed, the one that appears first in PATH wins. Always verify with which java (Linux/Mac) or where java (Windows).

Your First Java Program: Hello World

Every programmer starts here. Below is the classic, absolute minimum code required to print a message to the screen in Java. This code serves as the universal test that your JDK installation is working correctly.

JavaHelloWorld.java
public class HelloWorld {
    public static void main(String[] args) {
        System.out.println("Hello, World!");
    }
}

Line-by-Line Breakdown for Beginners

To an absolute beginner, the code above might look like a magical spell. Let's break it down line by line to understand exactly what each keyword does:

  • public (Access Modifier): Tells Java that this class is accessible and visible to any other class in the application, including the JVM itself.
  • class (Keyword): Used to declare that we are defining a new class. In Java, all code must reside inside a class.
  • HelloWorld (Name): The name of our class. Critical Rule: The name of the public class must match the name of the file exactly. Thus, this class must be saved in a file named HelloWorld.java. Java is case-sensitive, so helloworld or HELLOWORLD will fail to compile.
  • public static void main(String[] args): This is the Entry Point of your Java program. When you tell Java to run your program, the JVM looks specifically for this exact line to start execution.
    • public: Visible to the execution engine.
    • static: Allows the JVM to invoke this main method directly without having to create an instance of the class first.
    • void: Means this method does not return any value back after finishing.
    • main: The name of the method. The JVM is hardcoded to look for this name.
    • String[] args: A parameters array that allows passing input arguments to your program from the terminal.
  • System.out.println("..."): This is the command used to write output to the console.
    • System: A built-in system class that contains utility members.
    • out: An output stream connected to the console.
    • println: Short for "print line". Prints the string inside quotes, then moves the cursor to the next line.

The Manual Compile-Run Cycle

Before using an IDE (like IntelliJ IDEA) that hides the compilation process behind a "Run" button, every developer must understand the raw compile-run cycle. This cycle is the exact pipeline that runs under the hood of every system:

javac HelloWorld.java
# Result: This generates a new file in the directory named "HelloWorld.class"

# Step 2: Run the compiled bytecode using the JVM
# NOTE: DO NOT write ".class" or ".java" at the end here!
java HelloWorld
# Output: Hello, World!

Under the Hood: How the JVM Executes Hello World

An expert-level reference detailing JVM internals from class load to execution.

1. The Class Loader Subsystem

When you run java HelloWorld, the JVM starts and invokes the Class Loader. This subsystem performs three key phases sequentially:

  1. Loading: The Class Loader reads the binary data of HelloWorld.class and generates its corresponding binary representation in the Method Area of JVM memory.
  2. Linking:
    • Verification: Extremely critical phase. The Bytecode Verifier checks if the compiler output complies with JVM specifications. This guards against modified/malicious bytecode that could bypass language safety boundaries.
    • Preparation: Allocates memory for static variables and initializes them to default values.
    • Resolution: Replaces symbolic references in the bytecode with direct memory references.
  3. Initialization: Executes the static initializers of the class and assigns initial static values.

2. Execution Engine & JIT Compiler

Once loaded, the Execution Engine reads the bytecode instructions. It does this via two mechanisms:

  • Interpreter: Reads bytecode instructions one by one and translates them directly to OS-level instructions. This starts immediately but is slower.
  • JIT (Just-In-Time) Compiler: Profiles running code. If it identifies a method that is executed frequently ("hot spot"), it compiles that bytecode block directly into native machine code for the host CPU. Subsequent execution runs at native speed.

3. Advanced Hello World Reference Snippet

For experts and command-line automation, here is a complete program illustrating command-line inputs, system variable reads, and basic loops:

JavaHelloWorldAdvanced.java
public class HelloWorldAdvanced {
    public static void main(String[] args) {
        System.out.println("Hello, World!");
        System.out.printf("Java Version: %s%n", System.getProperty("java.version"));
        System.out.printf("Arguments Count: %d%n", args.length);
        for (int i = 0; i < args.length; i++) {
            System.out.printf("  → arg[%d]: %s%n", i, args[i]);
        }
    }
}

Real-World: Enter Build Tools

While understanding javac and java is crucial for interviews and debugging, no professional Java developer compiles code manually. Managing a massive classpath with hundreds of external library JARs manually is impossible.

In the real world, you will use a Build Tool. The two industry standards are Maven and Gradle.

  • Dependency Management: You tell the tool "I need Spring Boot and Jackson", and it automatically downloads those JARs and all their dependencies from a central repository.
  • Standardized Lifecycle: Running mvn clean install or ./gradlew build will automatically compile the code, run all unit tests, package the result into a deployable JAR file, and flag any errors.
  • Consistency: Build tools ensure that a project builds exactly the same way on your laptop, your coworker's laptop, and the CI/CD pipeline server.
ℹ️
We have a dedicated chapter on Build Tools later in the course, but it's important to know early on that manual compilation is just the foundation.

IDE Setup: IntelliJ IDEA

  • Project SDK: File → Project Structure → Project → SDK — always set explicitly, never "No SDK" or "default".
  • Language level: File → Project Structure → Project → Language Level — should match your SDK.
  • Auto-import: Settings → Editor → General → Auto Import — enable "Add unambiguous imports on the fly".
  • Save on focus lost: Settings → Appearance → System Settings → Autosave — enable for faster feedback.
  • Run configuration: Right-click main() → Run. Or create a Run Configuration for complex projects.
TIP
In IntelliJ, Shift+Shift (Search Everywhere) and Ctrl+Alt+L (Reformat Code) are the two most productive shortcuts to learn first.

Common Errors and Fixes

'javac' not recognized
PATH missing
JAVA_HOME\bin is not in PATH. Add it and restart the terminal. Run where javac to diagnose.
Error: Could not find or load main class
Class name wrong
Run java ClassName not java ClassName.class. Class name must match filename exactly including case.
has been compiled by a more recent version
Version mismatch
JDK that compiled the code is newer than the JRE running it. Ensure both are the same version.
UnsupportedClassVersionError
Bytecode too new
Class was compiled for Java 21 (class version 65) but JVM is Java 17 (class version 61). Upgrade the JVM or compile with --release flag.

Interview Preparation

Question 1
What is the Classpath?
The classpath tells the JVM and compiler where to look for compiled class files and JARs. Set with -cp flag or CLASSPATH env variable. In modern projects, build tools (Maven/Gradle) manage the classpath automatically.
Question 2
What is JIT compilation?
Just-In-Time compilation converts hot bytecode paths into native machine code at runtime. The JVM starts interpreting, profiles execution, then compiles frequently executed methods to native — giving Java near-native performance after a short warm-up period.
Question 3
JDK vs JRE vs JVM?
JVM: executes bytecode. JRE: JVM + standard library (deprecated since Java 11). JDK: JRE + compiler (javac) + developer tools. As a developer, you always install the JDK.
Question 4
What happens if you run 'java MyClass.class'?
It throws ClassNotFoundException: MyClass.class. The java command expects a fully-qualified class name (e.g., com.example.MyClass), not a filename. The .class extension is a file extension, not part of the class name.
