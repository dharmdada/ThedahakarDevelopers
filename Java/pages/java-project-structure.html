Theory Deep Dive

Structure as long-term maintainability

Mental Model

Project structure is an architectural decision. It decides where behavior lives, how modules depend on each other, how tests mirror production code, and how quickly a new developer can understand the system.

Layered folders are useful only if dependencies point in the right direction. Controllers should not contain business rules. Domain logic should not depend on HTTP. Repository code should not leak database details into the public API.

As projects grow, package structure should reflect business capability as much as technical layer. A billing package containing invoice, payment, and tax behavior is often easier to maintain than global controller, service, and repository folders containing every feature.

Principles That Matter

  • Follow standard directory layouts (e.g., src/main/java, src/test/java). Don't invent your own.
  • Organize packages by feature (com.app.user, com.app.order) rather than by layer (com.app.controllers, com.app.services).
  • Keep configuration files (like application.properties) in src/main/resources.

Common Misunderstandings

  • Putting all classes in the default (unnamed) package. This makes them invisible to classes in named packages.
  • Mixing test code and production code in the same directory.
  • Not using a unique package name (like a reversed domain name: com.yourcompany.project).

Mastery Checklist

  • You can set up a standard Maven/Gradle project structure from scratch.
  • You understand the difference between the src/ and target/ (or build/) directories.
  • You know where to place static assets and property files.

Standard Layout (Why It Matters)

A good folder structure is not cosmetics. It controls build speed, test isolation, deployment reliability, and onboarding speed for new developers.

  • Predictable location of code: Any developer can find business logic, tests, configs, and resources quickly.
  • Build tool compatibility: Maven and Gradle convention-over-configuration works best when folders follow standards.
  • Separation of concerns: Domain logic, API layer, infrastructure, and tests stay decoupled.

TIP
Keep package names by domain first, layer second when possible. Example: com.company.billing.invoice.service scales better than one giant service package for all modules.

Package Design Rules

Use reverse domain: com.company.product.

High
Place classes that change together in the same package.
Coupling
Low
Minimize imports between unrelated modules.
Visibility
Tight
Default/package-private first, then broaden only if needed.

Layered Architecture (Practical Mapping)

  • Controller/API layer: input validation, request/response mapping, status codes.
  • Service layer: business rules, transaction boundaries, orchestration.
  • Repository layer: persistence abstraction, SQL/ORM integration.
  • Domain layer: entities, value objects, invariants.
  • Infrastructure layer: external APIs, message brokers, storage adapters.

Do not leak persistence types into API contracts. Keep DTOs and entities separate to avoid accidental data exposure.

Quality Checklist for Every New Module

  • Module has its own package root and dedicated tests.
  • Config values are externalized in properties or environment variables.
  • Public classes have explicit purpose; dead utility classes are avoided.
  • Exceptions are translated at boundaries, not leaked raw to clients.
  • README includes build, run, and test commands.

Interview Preparation

Question 1
Why do we use reversed domain names for packages?
To prevent naming collisions. If two different companies create a 'User' class, com.apple.User and com.microsoft.User can coexist perfectly in the same application.
Question 2
What goes in src/main/resources?
Non-Java files needed at runtime, such as configuration properties, XML files, templates, and static images. They are bundled into the root of the resulting JAR.
