Theory Deep Dive

Collections as algorithmic choices

Mental Model

Collection classes encode algorithmic tradeoffs. Hash-based structures optimize lookup by hash distribution, tree-based structures maintain sorted order, array-backed structures optimize indexed access, and linked structures optimize some insert/remove patterns.

A collection API should reveal what the caller may rely on. If ordering matters, return List or SortedSet. If uniqueness matters, use Set. If key lookup matters, use Map. Vague types and undocumented contracts create hidden bugs.

Performance discussions should include data size, operation frequency, memory overhead, and mutation pattern. Big-O is a starting point, not a complete performance guarantee.

Beginner-Friendly, Expert-Useful Lens

If you are new: Beginners should connect each collection to a simple use case: list, set, map, queue, stack, sorted data.

If you are revising: Experts should refresh hash flooding, tree bins, fail-fast iterators, comparator consistency, memory overhead, and concurrent alternatives.

Mastery check: You understand Collections Deep Dive when you can explain the simple mental model, write a tiny runnable example, identify the common trap, and describe how it behaves in a real project.

Principles That Matter

  • Choose the right data structure for the job (e.g., ArrayList for fast reads, HashSet for uniqueness).
  • Understand time complexity (Big O notation) of basic operations like add, contains, and remove.
  • Use the Map interface for key-value pairs; it does not inherit from Collection.

Common Misunderstandings

  • Using LinkedList thinking it is faster than ArrayList. For modern hardware caching, ArrayList is almost always faster.
  • Forgetting to override equals() and hashCode() when using custom objects as keys in a HashMap or elements in a HashSet.
  • Modifying a collection during iteration using a normal for-each loop.

Mastery Checklist

  • You can explain the internal working mechanism of a HashMap.
  • You know the difference between fail-fast and fail-safe iterators.
  • You can correctly override equals() and hashCode() according to the contract.

Hard-Won Tips

  • HashMap iteration order is not a contract. If order matters, choose LinkedHashMap, TreeMap, or sort explicitly.
  • Comparator inconsistent with equals can make sorted sets and maps appear to lose or merge values unexpectedly.
  • Fail-fast iterators detect some concurrent modification bugs but are not a synchronization guarantee.
  • The load factor balances memory and collision cost. Default settings are good, but huge maps may need deliberate sizing.
  • A collection API should clarify ownership: can callers mutate it, retain it, or rely on its ordering?

List Implementations: The Ordered Sequence

A List is exactly what it sounds like: an ordered sequence of elements. You can have duplicates, and you access items by their exact position (index 0, 1, 2...).

The choice between ArrayList and LinkedList is a classic computer science trade-off:

  • ArrayList: Under the hood, this is just a standard array. If it gets full, Java silently creates a bigger one and copies the data over. It is blazing fast for reading data randomly (O(1)), but slow if you want to insert a value into the middle (because every subsequent element must be shifted over).
  • LinkedList: Elements are scattered across memory, with each element holding a pointer to the next one (like a scavenger hunt). It is extremely fast to insert or delete elements in the middle (just change the pointers), but very slow to read a random element (you have to traverse from the beginning every time).
Implementationget(i)add(end)add(mid)removeBest For
ArrayListO(1)O(1)*O(n)O(n)Random access, most use cases
LinkedListO(n)O(1)O(1)O(1)Frequent insert/remove at ends
CopyOnWriteArrayListO(1)O(n)O(n)O(n)Concurrent reads, rare writes
JavaList Operations
import java.util.*;

public class Main {
    public static void main(String[] args) {
        List<String> mutable = new ArrayList<>(List.of("a", "b"));
        List<String> immutable = List.of("x", "y");
        List<String> copy = List.copyOf(mutable);

        mutable.add("c");
        mutable.add(0, "first");
        mutable.set(1, "replaced");
        mutable.remove("c");
        mutable.remove(0);

        System.out.println("Contains a? " + mutable.contains("a"));
        System.out.println("Index of a: " + mutable.indexOf("a"));
        System.out.println("Immutable: " + immutable);
        System.out.println("Copy: " + copy);

        mutable.sort(Comparator.naturalOrder());
        mutable.sort(Comparator.comparing(String::length).reversed());
        System.out.println(mutable);
    }
}

Set Implementations: Uniqueness Guarantee

A Set is a collection that mathematically guarantees no duplicate elements. If you try to add "Apple" to a Set that already contains "Apple", the Set will simply ignore you.

The most common implementation is HashSet. It achieves O(1) lightning-fast performance by utilizing the hashCode() method to instantly calculate exactly where an object belongs in memory, completely eliminating the need to search.

ImplementationOrderadd/remove/containsBest For
HashSetNoneO(1)Fastest, general purpose
LinkedHashSetInsertion orderO(1)Ordered + fast
TreeSetSorted (natural/comparator)O(log n)Sorted unique values
EnumSetEnum orderO(1)Sets of enum values
JavaSet Operations
import java.util.*;

public class Main {
    public static void main(String[] args) {
        Set<String> a = new HashSet<>(Set.of("A", "B", "C"));
        Set<String> b = Set.of("B", "C", "D");

        Set<String> union = new HashSet<>(a);
        union.addAll(b);

        Set<String> intersection = new HashSet<>(a);
        intersection.retainAll(b);

        Set<String> difference = new HashSet<>(a);
        difference.removeAll(b);

        System.out.println("Union: " + union);
        System.out.println("Intersection: " + intersection);
        System.out.println("Difference: " + difference);
    }
}

Map Implementations: The Dictionary

A Map (often called a Dictionary in other languages) does not store single values; it stores Key-Value pairs. You provide a unique Key to look up a specific Value.

Think of a phone book: The Key is the person's name (which must be unique), and the Value is their phone number. If you know the exact name, finding the number takes O(1) time using a HashMap.

ImplementationOrderget/putBest For
HashMapNoneO(1)Fastest, general purpose
LinkedHashMapInsertion orderO(1)Ordered + fast
TreeMapSorted by keyO(log n)Sorted keys
ConcurrentHashMapNoneO(1)Thread-safe
EnumMapEnum orderO(1)Enum keys
JavaMap Operations
import java.util.*;

public class Main {
    public static void main(String[] args) {
        Map<String, Integer> map = new HashMap<>();
        map.put("Alice", 25);
        map.put("Bob", 30);

        int age = map.getOrDefault("Charlie", 0);
        map.putIfAbsent("Alice", 99);
        map.computeIfAbsent("Dave", key -> key.length());
        map.merge("Alice", 5, Integer::sum);

        map.forEach((key, value) -> System.out.println(key + ": " + value));
        System.out.println("Charlie age: " + age);

        Map<String, Integer> immutable = Map.of("A", 1, "B", 2);
        Map<String, Integer> immutable2 = Map.ofEntries(Map.entry("A", 1), Map.entry("B", 2));
        System.out.println(immutable);
        System.out.println(immutable2);
    }
}

Under the Hood: HashMap Load Factor

A HashMap stores data in an array of "buckets". When you create a HashMap, it starts with an initial capacity (default 16 buckets). But what happens if you add 17 items?

The HashMap uses a Load Factor (default 0.75). This means when the map is 75% full (12 items), it triggers a massive operation called Rehashing. It creates a brand new array double the size (32 buckets) and recalculates the hash position for every single existing element. Rehashing is a huge performance hit. If you know you are going to put 100,000 items in a map, always initialize it with a large capacity: new HashMap<>(135000); to prevent rehashing.

Queue & Deque: Line Up

A Queue represents a line at a grocery store. It enforces FIFO (First-In, First-Out). The first element added to the queue is the first one processed.

A Deque (Double Ended Queue) is a line where you can add or remove from either the front or the back. Because of this, it is the standard class used to implement Stacks (LIFO - Last-In, First-Out), like a stack of plates.

Java
import java.util.*;

public class Main {
    public static void main(String[] args) {
        Queue<String> queue = new LinkedList<>();
        queue.offer("First");
        queue.offer("Second");
        System.out.println(queue.peek());
        System.out.println(queue.poll());

        PriorityQueue<Integer> pq = new PriorityQueue<>();
        pq.offer(30);
        pq.offer(10);
        pq.offer(20);
        System.out.println("Priority poll: " + pq.poll());

        Deque<String> deque = new ArrayDeque<>();
        deque.push("A");
        deque.push("B");
        System.out.println("Stack pop: " + deque.pop());

        deque.offerFirst("Front");
        deque.offerLast("Back");
        System.out.println(deque);
    }
}
💡
Don't Use Stack ClassThe legacy java.util.Stack class is synchronized and slow. Use ArrayDeque as your stack instead — it's faster and more correct.

Generics & Wildcards: PECS

Generics (<T>) allow you to write classes and methods that work with any object type, while still providing strict compile-time type safety.

Wildcards (?) are used when dealing with generic collections. The golden rule of wildcards is PECS (Producer Extends, Consumer Super). If you only need to read from a collection, use extends. If you only need to write to it, use super.

JavaWildcards
import java.util.*;

public class Main {
    public static double sum(List<? extends Number> list) {
        return list.stream().mapToDouble(Number::doubleValue).sum();
    }

    public static void addNumbers(List<? super Integer> list) {
        list.add(1);
        list.add(2);
    }

    public static <T extends Comparable<T>> T max(T a, T b) {
        return a.compareTo(b) >= 0 ? a : b;
    }

    public static void main(String[] args) {
        System.out.println(sum(List.of(1, 2, 3)));
        System.out.println(sum(List.of(1.5, 2.5)));

        List<Number> numbers = new ArrayList<>();
        addNumbers(numbers);
        System.out.println(numbers);

        System.out.println(max("apple", "banana"));
    }
}

Comparable vs Comparator

Java
import java.util.*;

record Employee(String name, double salary) implements Comparable<Employee> {
    @Override
    public int compareTo(Employee other) {
        return Double.compare(this.salary, other.salary);
    }
}

public class Main {
    public static void main(String[] args) {
        List<Employee> employees = new ArrayList<>(List.of(
            new Employee("Bob", 70000),
            new Employee("Alice", 90000),
            new Employee("Alice", 80000)
        ));

        Collections.sort(employees);
        System.out.println("By natural salary: " + employees);

        Comparator<Employee> byNameThenSalary = Comparator
            .comparing(Employee::name)
            .thenComparingDouble(Employee::salary);

        employees.sort(byNameThenSalary);
        System.out.println("By name then salary: " + employees);
    }
}

Real-World Use Case: Task Schedulers

Imagine you are building a print job spooler or a background task processor in a server. Tasks arrive with different priorities: some are urgent system alerts, and others are low-priority email notifications.

A PriorityQueue is the perfect data structure for this. Instead of processing tasks in the exact order they arrive (FIFO), the PriorityQueue ensures that the highest-priority task is always at the head of the queue, ready to be processed next, regardless of when it arrived.

Interview Preparation

Question 1
How does a HashMap work internally?
It uses an array of Nodes. It computes the hash code of the key to find the array index. If there's a collision, it uses a linked list (or Red-Black tree in Java 8+) at that index.
Question 2
ArrayList vs LinkedList?
ArrayList uses a dynamic array (fast random access O(1), slow insertions/deletions in middle O(n)). LinkedList uses a doubly-linked list (slow random access O(n), fast insertions/deletions O(1) if node is known).
