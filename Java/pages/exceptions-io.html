Exceptions and Error Handling

Java's structured failure mechanism — how to detect, report, propagate, and recover from failures without crashing your program.

TH

Theory Deep Dive

Failure as a first-class citizen in your API contract

Mental Model

Every method makes an implicit promise: "I will do X if you give me valid input and the environment cooperates." When that promise can't be kept, the method must communicate failure — and that communication is the exception system.

Java splits failures into two categories based on who is responsible for dealing with them:

  • Checked exceptions — External, environmental failures the caller can realistically recover from: file not found, network timeout, database down. The compiler forces you to acknowledge them. These belong in throws clauses.
  • Unchecked exceptions (RuntimeException) — Programming bugs: null pointer, array out of bounds, illegal argument. The caller cannot "recover" — the code must be fixed. These should NOT be caught unless you have a very specific reason.
  • Errors — JVM-level catastrophes: out of memory, stack overflow, class loading failure. You cannot and should not catch these. If they occur, the JVM itself is in trouble.

The key design question is not "should I use a checked or unchecked exception?" — it is "can the caller reasonably recover from this failure?" If yes → checked. If it's a bug → unchecked.

The Exception Hierarchy

ℹ️
Complete Throwable tree
Throwable
├── Error ← JVM failures. Don't catch. (OutOfMemoryError, StackOverflowError, AssertionError)
└── Exception
    ├── RuntimeException ← Unchecked. Fix the bug. (NullPointerException, IllegalArgumentException, ClassCastException, IndexOutOfBoundsException)
    └── Checked Exceptions ← Must handle or declare. (IOException, SQLException, ParseException, InterruptedException)

Principles That Matter

  • Catch what you can handle — If you catch an exception and don't know what to do, re-throw it. An empty catch block is the worst possible thing.
  • Never swallow exceptions silently — catch (Exception e) {} is a bug disguised as code. At minimum, log it.
  • Don't catch Exception or Throwable at low levels — Catch the most specific type possible. Catching Exception hides bugs.
  • Include context in exception messages — "User not found: id=" + id is infinitely more useful than "User not found".
  • Prefer unchecked for library APIs — Checked exceptions in widely-used APIs become a maintenance burden. Spring, Guava, and most modern Java frameworks use unchecked.

Common Misunderstandings

  • Exceptions are for exceptional cases — Do NOT use exceptions for normal control flow (e.g., throw to signal "item not in list"). Use Optional or a return value instead.
  • final vs finally vs finalize — final: variable/method/class modifier. finally: cleanup block. finalize(): deprecated GC hook — never use it.
  • Catching Exception masks NullPointerException — If you catch Exception everywhere, a NPE caused by your bug gets silently swallowed.
  • finally always runs — except when it doesn't — System.exit() or JVM crash before reaching finally will skip it.

Mastery Checklist

  • You can explain checked vs unchecked with the "can the caller recover?" test.
  • You can write multi-catch, finally, and try-with-resources blocks.
  • You can create a custom exception hierarchy for a domain.
  • You know when to wrap, re-throw, or translate exceptions at layer boundaries.
  • You can explain exception chaining (new RuntimeException("msg", cause)).

Try-Catch-Finally — Complete Anatomy

JavaAll exception handling forms
// ---- Basic try-catch ----
try {
    int result = 10 / 0;                  // throws ArithmeticException
} catch (ArithmeticException e) {
    System.out.println("Cannot divide by zero: " + e.getMessage());
}

// ---- Multi-catch (Java 7+) — one block for related types ----
try {
    riskyOperation();
} catch (IOException | SQLException e) {  // e is effectively final
    log.error("External failure: {}", e.getMessage(), e);
    throw new ServiceException("Operation failed", e); // wrap and rethrow
}

// ---- Try-catch-finally ----
Connection conn = null;
try {
    conn = getConnection();
    conn.execute(sql);
} catch (SQLException e) {
    log.error("DB error", e);
} finally {
    // Runs ALWAYS — whether exception occurred or not
    if (conn != null) conn.close();  // but use try-with-resources instead!
}

// ---- Exception chaining — preserve the root cause ----
try {
    Files.readString(Path.of("config.json"));
} catch (IOException e) {
    // Wrap checked IOException as unchecked, but keep the original cause
    throw new ConfigurationException("Cannot load config file", e);
}
// Later: e.getCause() returns the original IOException

// ---- Re-throw after partial handling ----
try {
    processPayment(order);
} catch (PaymentException e) {
    auditLog.record("PAYMENT_FAILED", order.id(), e.getMessage());
    throw e;  // re-throw — caller still needs to know it failed
}
WARN
Catch order matters. Always catch subclasses before superclasses. Catching Exception before IOException makes the IOException block unreachable — the compiler will flag it.

Try-With-Resources — Automatic Resource Cleanup

Any object implementing AutoCloseable (or Closeable) can be declared in the try header. Java guarantees close() is called in reverse declaration order, even if an exception is thrown.

JavaResource cleanup — the right way
// Single resource — FileReader auto-closed after block exits
try (var reader = new BufferedReader(new FileReader("data.txt"))) {
    String line;
    while ((line = reader.readLine()) != null) {
        System.out.println(line);
    }
} catch (IOException e) {
    log.error("Failed to read file", e);
}
// reader.close() called automatically — no finally block needed

// Multiple resources — closed in REVERSE order (out before in)
try (var in  = new FileInputStream("source.txt");
     var out = new FileOutputStream("dest.txt")) {
    in.transferTo(out);
}

// Custom AutoCloseable — any class can participate
class DatabaseSession implements AutoCloseable {
    public DatabaseSession() { System.out.println("Session opened"); }
    @Override public void close() { System.out.println("Session closed"); }
}

try (var session = new DatabaseSession()) {
    session.query("SELECT ...");
}  // "Session closed" printed automatically

// Suppressed exceptions — if both body AND close() throw,
// the body exception is primary; close() exception is suppressed
try (var r = new BrokenResource()) {
    throw new RuntimeException("body failed");
    // BrokenResource.close() also throws — that exception is suppressed
    // Access via: primaryException.getSuppressed()[0]
}
TIP
Everything that holds a system resource — file handles, DB connections, HTTP connections, thread pools — should be declared in try(). If you ever write a finally { resource.close(); }, you probably should be using try-with-resources instead.

Custom Exception Hierarchies

Well-designed systems define their own exception types that reflect the domain vocabulary, not just the technical failure.

JavaDomain exception hierarchy — e-commerce example
// Base domain exception — unchecked (modern approach)
public class DomainException extends RuntimeException {
    private final String errorCode;

    public DomainException(String errorCode, String message) {
        super(message);
        this.errorCode = errorCode;
    }

    public DomainException(String errorCode, String message, Throwable cause) {
        super(message, cause);
        this.errorCode = errorCode;
    }

    public String errorCode() { return errorCode; }
}

// Specific domain exceptions — each carries its own context
public class InsufficientFundsException extends DomainException {
    private final BigDecimal shortfall;

    public InsufficientFundsException(BigDecimal shortfall) {
        super("PAYMENT_INSUFFICIENT_FUNDS",
              "Payment declined: short by " + shortfall);
        this.shortfall = shortfall;
    }

    public BigDecimal shortfall() { return shortfall; }
}

public class ProductNotFoundException extends DomainException {
    public ProductNotFoundException(String productId) {
        super("PRODUCT_NOT_FOUND", "Product not found: " + productId);
    }
}

// Usage — expressive, testable, carries context
public Order checkout(Cart cart) {
    Product p = productRepo.findById(cart.productId())
        .orElseThrow(() -> new ProductNotFoundException(cart.productId()));

    if (wallet.balance().compareTo(p.price()) < 0) {
        throw new InsufficientFundsException(p.price().subtract(wallet.balance()));
    }
    return createOrder(p, cart);
}

// Caller handles domain exception — or lets it bubble to global handler
try {
    Order order = checkoutService.checkout(cart);
} catch (ProductNotFoundException e) {
    return ResponseEntity.notFound().build();
} catch (InsufficientFundsException e) {
    return ResponseEntity.status(402)
        .body(Map.of("shortfall", e.shortfall(), "code", e.errorCode()));
}

Production Exception Patterns

JavaLayer boundary translation — don't leak implementation details
// ❌ BAD: leaking JDBC internals through the service layer
public User findUser(long id) throws SQLException {  // caller knows it's JDBC!
    return jdbcRepo.findById(id);
}

// ✅ GOOD: translate at the repository boundary
public User findUser(long id) {
    try {
        return jdbcRepo.findById(id);
    } catch (SQLException e) {
        throw new DataAccessException("Failed to load user: " + id, e);
        // Keeps cause chain intact — e is wrapped, not lost
    }
}

// Global exception handler in Spring Boot
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(ProductNotFoundException.class)
    public ResponseEntity handleNotFound(ProductNotFoundException e) {
        return ResponseEntity.status(404)
            .body(new ErrorResponse(e.errorCode(), e.getMessage()));
    }

    @ExceptionHandler(DomainException.class)
    public ResponseEntity handleDomain(DomainException e) {
        log.warn("Domain error [{}]: {}", e.errorCode(), e.getMessage());
        return ResponseEntity.status(422)
            .body(new ErrorResponse(e.errorCode(), e.getMessage()));
    }

    @ExceptionHandler(Exception.class)
    public ResponseEntity handleUnexpected(Exception e) {
        log.error("Unexpected error", e);   // log FULL stack trace for bugs
        return ResponseEntity.status(500)
            .body(new ErrorResponse("INTERNAL_ERROR", "An unexpected error occurred"));
        // NEVER expose internal details (stack traces) to API consumers!
    }
}

Interview Preparation

Question 1
Checked vs Unchecked exceptions?
Checked: compiler-enforced, represent external failures caller can recover from (IOException, SQLException). Unchecked (RuntimeException): programming bugs, no recovery possible (NPE, IllegalArgumentException). The test: "Can the caller do something meaningful if this happens?"
Question 2
What if finally also throws?
The exception from finally replaces the original try-block exception — the original is lost. This is why you should never throw from finally. Try-with-resources handles this correctly via suppressed exceptions (getSuppressed()).
Question 3
Error vs Exception?
Error: JVM-level catastrophes (OutOfMemoryError, StackOverflowError) — should not be caught. Exception: application-level conditions that can potentially be handled. Both extend Throwable.
Question 4
Why prefer unchecked exceptions in modern Java?
Checked exceptions force callers to handle or declare failures at every call site, creating boilerplate and leaking implementation details. Modern frameworks (Spring, Hibernate) use unchecked to let exceptions propagate to a single centralized handler while keeping method signatures clean.
