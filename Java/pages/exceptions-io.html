Exceptions & I/O

Error handling, file operations, and resource management.

TH

Theory Deep Dive

Failure as part of the API contract

Mental Model

Exceptions are Java's structured mechanism for reporting failure. A method signature and documentation should make it clear which failures callers can handle and which indicate programming errors or unrecoverable conditions.

Checked exceptions force callers to acknowledge recoverable external failures such as I/O. Unchecked exceptions are better for invalid arguments, illegal state, and programming defects. The boundary between them should be chosen by caller responsibility, not personal preference.

I/O code must also manage resources. Files, sockets, streams, and database connections are outside normal heap memory and must be closed reliably. try-with-resources exists for this reason.

Principles That Matter

  • Understand the purpose of the feature before memorizing its syntax.
  • Connect the concept to runtime behavior, API design, and failure cases.
  • Prefer readable code that makes assumptions explicit.
  • Use the feature when it reduces complexity, not just because it exists.

Common Misunderstandings

  • Learning only the code pattern without the reason behind it.
  • Ignoring edge cases until they appear as bugs.
  • Using the feature everywhere instead of where it fits the problem.
  • Skipping the connection between this topic and surrounding Java concepts.

Mastery Checklist

  • You can explain exceptions io in plain language without reading code.
  • You can identify the right and wrong use cases for the concept.
  • You can debug common failures related to the topic.
  • You can teach the concept using a small real-world example.

The Exception Hierarchy: Anticipating Disaster

In the real world, things go wrong. Files get deleted, network connections drop, and users type text into number fields. Java forces you to anticipate these disasters using Exceptions.

There is a strict hierarchy of disasters in Java:

  • Errors: Catastrophic JVM failures (like running out of RAM or a Stack Overflow). Like a meteor striking the data center. You cannot and should not try to catch these.
  • Unchecked Exceptions (RuntimeException): Programming bugs (like a NullPointerException or an IndexOutOfBoundsException). You don't have to declare these because you should fix the bug instead of catching it.
  • Checked Exceptions: Anticipated external failures (like a file missing or a network timeout). The Java compiler forces you to write a safety net (a try-catch block) for these, or the code won't even compile.
ℹ️
Exception Tree Throwable
├── Error (JVM errors — don't catch these: OutOfMemoryError, StackOverflowError)
└── Exception
    ├── RuntimeException (Unchecked — NullPointerException, ArrayIndexOutOfBoundsException, etc.)
    └── Checked Exceptions (IOException, SQLException — MUST be handled or declared)

Try-Catch-Finally: The Safety Net

When you call a method that might throw an exception (like reading a file), you place it inside a try block. If the method explodes, it throws an Exception object down to the catch block, which acts as a safety net.

The finally block is the "clean-up crew". Whether the try block succeeded, or whether the catch block fired, the finally block is guaranteed to run. It is traditionally used to close open files or database connections.

Java
try {
    int result = 10 / 0;
} catch (ArithmeticException e) {
    System.out.println("Cannot divide by zero: " + e.getMessage());
} catch (Exception e) {
    System.out.println("General error");
} finally {
    System.out.println("Always runs");
}

// Multi-catch (catch multiple types in one block)
try {
    // risky code
} catch (IOException | SQLException e) {
    System.out.println("IO or SQL error: " + e.getMessage());
}

Try-With-Resources: The Automated Janitor

Developers used to forget to close files in the finally block, causing massive memory leaks. Java 7 introduced Try-With-Resources.

If you declare a resource inside the parentheses of a try block, Java acts as an automated janitor. As soon as the try block finishes (or crashes), Java will automatically close the file for you. The resource just needs to implement the AutoCloseable interface.

Java
// Resources that implement AutoCloseable are auto-closed
try (var reader = new BufferedReader(new FileReader("file.txt"))) {
    String line;
    while ((line = reader.readLine()) != null) {
        System.out.println(line);
    }
} // reader.close() called automatically, even if exception occurs!

// Multiple resources
try (
    var in = new FileInputStream("input.txt");
    var out = new FileOutputStream("output.txt")
) {
    in.transferTo(out); // Copy file!
}

Custom Exceptions

Java
// Checked exception (must be declared with 'throws')
public class InsufficientFundsException extends Exception {
    private final double amount;
    public InsufficientFundsException(double amount) {
        super("Short by: " + amount);
        this.amount = amount;
    }
    public double getAmount() { return amount; }
}

// Unchecked exception (no need to declare)
public class InvalidAgeException extends RuntimeException {
    public InvalidAgeException(String msg) { super(msg); }
}

// Usage
public void withdraw(double amount) throws InsufficientFundsException {
    if (amount > balance) throw new InsufficientFundsException(amount - balance);
    balance -= amount;
}

Modern File I/O (NIO.2)

In older versions of Java, reading a file required chaining together massive, ugly streams (new BufferedReader(new InputStreamReader(new FileInputStream...))). It was a nightmare for beginners.

Java's NIO.2 (New I/O) API modernized this. By using the Path and Files utility classes, you can read, write, copy, and delete files with single-line commands.

Javajava.nio.file
import java.nio.file.*;

Path path = Path.of("data", "example.txt");

// Write
Files.writeString(path, "Hello World");
Files.write(path, List.of("Line 1", "Line 2"));

// Read
String content = Files.readString(path);
List<String> lines = Files.readAllLines(path);

// Stream lines (lazy — good for huge files)
try (Stream<String> stream = Files.lines(path)) {
    stream.filter(l -> l.contains("error")).forEach(System.out::println);
}

// File operations
Files.exists(path);
Files.createDirectories(Path.of("a/b/c"));
Files.copy(source, target, StandardCopyOption.REPLACE_EXISTING);
Files.move(source, target);
Files.delete(path);

// Walk directory tree
try (Stream<Path> walk = Files.walk(Path.of("."))) {
    walk.filter(p -> p.toString().endsWith(".java"))
        .forEach(System.out::println);
}

Real-World Use Case: Defensive Programming

In web servers (like Spring Boot applications), exception handling is used to prevent the server from crashing when a user sends invalid data.

Instead of letting a NullPointerException or DatabaseException bubble up and show a ugly stack trace to the user, developers use a "Global Exception Handler". This component catches all exceptions, logs the error details for the developers to see, and returns a clean, friendly error message (like "Service temporarily unavailable") to the end user with an appropriate HTTP status code.

Interview Preparation

Question 1
Checked vs Unchecked Exceptions?
**Checked exceptions** (e.g., IOException) are checked at compile-time. You MUST handle them using try-catch or declare them with throws. **Unchecked exceptions** (e.g., NullPointerException) happen at runtime and do not force handle-or-declare.
Question 2
What is Try-with-Resources?
Introduced in Java 7, it is a try statement that declares one or more resources. The resource is an object that must be closed after the program is finished with it (implements AutoCloseable). It guarantees that the resource will be closed at the end of the statement.
Question 3
Error vs Exception?
**Errors** (like OutOfMemoryError) are serious problems that a reasonable application should not try to catch. **Exceptions** indicate conditions that a reasonable application might want to catch.
