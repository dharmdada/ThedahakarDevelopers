Generics Deep Dive (<T>)

Type-safe APIs without code duplication. Type Erasure, Wildcards, PECS, and real-world generic design.

TH

Theory Deep Dive

Type parameters as compile-time safety without runtime overhead

Mental Model

Before Java 5 (2004), all collections held raw Object references. Every retrieval required a cast, and if you put the wrong type in, you only discovered the bug at runtime as a ClassCastException. Generics moved type checking from runtime to compile time.

A List<String> and a List<Integer> share the same class file at runtime — they are both ArrayList. The generic information is used by the compiler to insert safe casts and reject invalid additions, then erased. This is type erasure.

Wildcards (?) describe flexibility at API boundaries without giving up safety. The PECS rule — Producer Extends, Consumer Super — is the single most important rule for designing generic method parameters correctly.

Beginner-Friendly, Expert-Useful Lens

If you are new: Beginners should learn that List means the list should only contain strings.

If you are revising: Experts should refresh type erasure, variance, wildcards, bridge methods, heap pollution, and API design constraints.

Mastery check: You understand Generics when you can explain the simple mental model, write a tiny runnable example, identify the common trap, and describe how it behaves in a real project.

Principles That Matter

  • Generic type information exists at compile time only. At runtime, List<String> is just List.
  • List<Dog> is NOT a subtype of List<Animal> (generics are invariant). Use wildcards for flexibility.
  • Use PECS: extends when you only read, super when you only write.
  • Prefer generic methods over raw types for utility code (sort, min, max helpers).

Common Misunderstandings

  • Thinking List<String> is a subtype of List<Object> — It is NOT. If it were, List<Object> list = new ArrayList<String>() would let you add an Integer to a String list at runtime.
  • Trying to instantiate a type parameter — new T() fails because T is erased to Object at runtime. Pass a Class<T> or a Supplier<T> instead.
  • Creating generic arrays — new T[10] is illegal. Use new Object[10] with an unchecked cast, or use a List<T>.
  • Checking instanceof with type parameters — if (list instanceof List<String>) cannot work at runtime. Use raw type check: instanceof List<?>.

Mastery Checklist

  • You can write a generic class, generic method, and generic interface.
  • You understand PECS and can apply it without looking it up.
  • You can explain type erasure and its consequences.
  • You can use bounded type parameters (<T extends Comparable<T>>).
  • You can design a type-safe generic repository pattern.

Hard-Won Tips

  • Generics are mostly compile-time. At runtime, type erasure means List and List do not carry full element type information.
  • Use extends when you read from a producer and super when you write to a consumer: PECS is about data direction, not memorization.
  • Raw types are escape hatches from type safety. They often move errors from compile time to runtime.
  • Generic arrays are restricted because arrays know their component type at runtime while generics are erased.
  • A clean generic API makes invalid combinations impossible without forcing callers into casts.

Type Erasure: The Grand Illusion

Generics are a compile-time feature. The compiler checks types, then erases them to maintain binary compatibility with pre-Java-5 code. The JVM has no idea about <T>:

JavaWhat you write vs what the JVM runs
import java.util.ArrayList;
import java.util.List;

public class Main {
    public static void main(String[] args) {
        // What you write:
        List<String> list = new ArrayList<>();
        list.add("Hello");
        String s = list.get(0);

        // What the JVM actually runs (after erasure):
        // List list = new ArrayList();           // raw type
        // list.add("Hello");
        // String s = (String) list.get(0);      // compiler inserts invisible cast

        // Runtime type check — both print "class java.util.ArrayList":
        System.out.println(new ArrayList<String>().getClass());
        System.out.println(new ArrayList<Integer>().getClass()); // SAME class!
    }
}
Cannot do
new T()
T is erased to Object. Solution: pass a Supplier<T> or Class<T> factory.
Cannot do
new T[10]
Generic array creation is illegal. Use List<T> or suppress with Object[] cast.
Cannot do
x instanceof T
T is erased at runtime. Use instanceof List<?> or pass Class<T> and call isInstance().
Cannot do
static T field
Static fields are shared by all instances. A static T field would be ambiguous for List<String> and List<Integer>.

The Danger of Heap Pollution

Because of Type Erasure, Java allows you to mix "Raw Types" (like List) with Parameterized Types (like List<String>) for backward compatibility. But doing this causes a dangerous bug known as Heap Pollution.

JavaHeap Pollution Example
import java.util.ArrayList;
import java.util.List;

public class Main {
    public static void main(String[] args) {
        List<String> strings = new ArrayList<>();
        List rawList = strings; // Allowed for legacy code
        rawList.add(42); // We just put an Integer into a List<String>!

        // The compiler doesn't stop this, but at runtime:
        try {
            String s = strings.get(0); // 💥 ClassCastException! Integer cannot be cast to String
        } catch (ClassCastException e) {
            System.out.println("Caught: " + e.getMessage());
        }
    }
}
⚠️
Never use Raw Types in modern Java. Always parameterize your collections, even if it's just List<Object>, to let the compiler enforce safety.

Generic Classes and Methods

JavaGeneric class: type-safe Pair
import java.util.Objects;

class Pair<A, B> {
    private final A first;
    private final B second;

    public Pair(A first, B second) {
        this.first  = first;
        this.second = second;
    }

    public A first()  { return first; }
    public B second() { return second; }

    public Pair<B, A> swap() { return new Pair<>(second, first); }

    public static <X, Y> Pair<X, Y> of(X x, Y y) { return new Pair<>(x, y); }

    @Override
    public String toString() { return "(" + first + ", " + second + ")"; }
}

public class Main {
    public static void main(String[] args) {
        // Usage — fully type-safe, no casts
        Pair<String, Integer> p = Pair.of("Alice", 42);
        String name                  = p.first();   // String
        int    age                   = p.second();  // int (auto-unboxed)
        Pair<Integer, String> flipped = p.swap();
        System.out.println(p + " flipped: " + flipped);
    }
}
JavaGeneric methods with bounded type parameters
import java.util.ArrayList;
import java.util.List;

public class Main {
    // T must implement Comparable — allows calling compareTo()
    public static <T extends Comparable<T>> T max(T a, T b) {
        return a.compareTo(b) >= 0 ? a : b;
    }

    // Generic method returning a new collection
    public static <T> List<T> repeat(T element, int times) {
        List<T> result = new ArrayList<>(times);
        for (int i = 0; i < times; i++) result.add(element);
        return result;
    }

    public static void main(String[] args) {
        String largest = max("apple", "banana");   // "banana"
        int    bigger  = max(10, 20);              // 20

        List<String>  words  = repeat("Java", 3);   // ["Java", "Java", "Java"]
        List<Integer> zeros  = repeat(0, 5);         // [0, 0, 0, 0, 0]
        System.out.println(largest + ", " + bigger + ", " + words + ", " + zeros);
    }
}

Wildcards and PECS — The Most Important Generic Rule

Generics are invariant: List<Dog> is not a List<Animal>. Wildcards provide controlled flexibility:

JavaPECS in complete, commented action
import java.util.*;

class Animal {
    double weightKg() { return 0; }
}
class Dog extends Animal {
    @Override double weightKg() { return 12.5; }
}
class Cat extends Animal {
    @Override double weightKg() { return 4.5; }
}

public class Main {
    public static double totalWeight(List<? extends Animal> animals) {
        double total = 0;
        for (Animal a : animals) {
            total += a.weightKg();
        }
        return total;
    }

    public static void addDogs(List<? super Dog> shelter, int count) {
        for (int i = 0; i < count; i++) {
            shelter.add(new Dog());
        }
    }

    public static void main(String[] args) {
        System.out.println(totalWeight(List.of(new Dog(), new Dog())));
        System.out.println(totalWeight(List.of(new Cat())));

        List<Animal> shelter = new ArrayList<>();
        addDogs(shelter, 2);
        System.out.println("Shelter size: " + shelter.size());
    }
}
TIP
PECS mnemonic: Think about the method's perspective. If the collection gives you data to read → use extends. If the collection receives data you add → use super. If you do both → use a plain type parameter T.

Real-World Generic Design Patterns

Generic Repository Interface

JavaSpring-Data-style repository pattern
import java.util.*;

interface Repository<T, ID> {
    Optional<T> findById(ID id);
    List<T> findAll();
    T save(T entity);
    void delete(ID id);
}

record User(Long id, String name) {}
record Product(UUID id, String name) {}
record Order(Long id, Long customerId) {}

interface UserRepository extends Repository<User, Long> {}
interface ProductRepository extends Repository<Product, UUID> {}
interface OrderRepository extends Repository<Order, Long> {
    List<Order> findByCustomerId(Long customerId);
}

class InMemoryUserRepository implements UserRepository {
    private final Map<Long, User> users = new HashMap<>();

    public Optional<User> findById(Long id) { return Optional.ofNullable(users.get(id)); }
    public List<User> findAll() { return new ArrayList<>(users.values()); }
    public User save(User user) { users.put(user.id(), user); return user; }
    public void delete(Long id) { users.remove(id); }
}

public class Main {
    public static void main(String[] args) {
        UserRepository repo = new InMemoryUserRepository();
        repo.save(new User(1L, "Alice"));
        System.out.println(repo.findById(1L).orElseThrow());
    }
}

Generic Result Type (Error Handling)

JavaType-safe Result wrapping success or failure
sealed interface Result<T> permits Result.Success, Result.Failure {
    record Success<T>(T value) implements Result<T> {}
    record Failure<T>(String error) implements Result<T> {}

    static <T> Result<T> ok(T value) { return new Success<>(value); }
    static <T> Result<T> err(String msg) { return new Failure<>(msg); }

    default T getOrElse(T fallback) {
        return this instanceof Success<T> s ? s.value() : fallback;
    }
}

record User(String email) {}

public class Main {
    static Result<User> findByEmail(String email) {
        return email.endsWith("@example.com")
            ? Result.ok(new User(email))
            : Result.err("User not found");
    }

    public static void main(String[] args) {
        Result<User> result = findByEmail("alice@example.com");
        switch (result) {
            case Result.Success<User> s -> System.out.println("Found: " + s.value());
            case Result.Failure<User> f -> System.out.println("Error: " + f.error());
        }
    }
}

Interview Preparation

Question 1
What is Type Erasure?
The compiler removes all generic type information after type-checking, replacing type parameters with Object (or the first bound). The JVM sees no <T> at runtime. This maintains binary compatibility with pre-Java 5 code but restricts: no new T(), no instanceof T, no generic arrays.
Question 2
What is the PECS Principle?
Producer Extends, Consumer Super. If a collection produces elements you read out → use ? extends T. If a collection consumes elements you put in → use ? super T. See Collections.copy(dest, src) for the canonical example.
Question 3
Is List<String> a subtype of List<Object>?
No — generics are invariant. If they were covariant, list.add(42) on a variable typed List<Object> but actually pointing to a List<String> would cause a runtime ClassCastException. The invariance is a type-safety guarantee. Use List<? extends Object> or List<?> for read-only flexibility.
Question 4
How do you instantiate a generic type T?
You cannot write new T() due to erasure. Patterns: (1) Pass a Supplier<T> factory: supplier.get(). (2) Pass a Class<T> and call clazz.getDeclaredConstructor().newInstance() via reflection. Spring uses option 2 internally for bean creation.
