Methods

Defining, calling, overloading, recursion, and how Java passes data.

TH

Theory Deep Dive

Methods as boundaries of meaning

Mental Model

A method is a named unit of behavior. It should take clear input, perform one coherent operation, and return or cause a clear result. Good methods reduce the amount of code a reader must hold in working memory.

Java passes arguments by value. For primitives, the value is the primitive itself. For objects, the value is the reference. This means a method cannot reassign the caller's variable, but it can mutate the object that the copied reference points to.

Overloading lets methods share a name when they represent the same concept with different inputs. It should not be used to hide unrelated behavior behind the same name.

Principles That Matter

  • A method should do one thing and do it well (Single Responsibility Principle).
  • Keep parameter lists short. If a method takes more than 3-4 arguments, consider passing an object.
  • Understand method signatures: name and parameter types. Return type is NOT part of the signature.

Common Misunderstandings

  • Confusing method overloading (same name, different parameters) with overriding (subclass replacing parent's method).
  • Thinking you can pass primitives by reference. Java is strictly pass-by-value.
  • Ignoring the return value of immutable object methods (e.g., calling str.toLowerCase() without assigning the result).

Mastery Checklist

  • You can explain method overloading and write an example.
  • You know how varargs (... syntax) works under the hood (it's an array).
  • You can differentiate between static and instance methods.

Defining & Calling Methods: The Verbs of Java

If classes and objects are the "nouns" of Java, then methods are the "verbs." A method is a reusable block of code that performs a specific task. By wrapping logic in a method, you avoid repeating the same code multiple times, making your application easier to maintain and test.

Every method defines a strict contract: it declares exactly what it needs as input (parameters) and exactly what it promises to give back (return type).

JavaMethod Anatomy
//  accessModifier  returnType  methodName(parameters)
    public          int        add(int a, int b) {
        return a + b;
    }

// void = no return value
public void greet(String name) {
    System.out.println("Hello " + name);
}

// Calling methods
int sum = add(3, 5);     // returns 8
greet("Alice");           // prints "Hello Alice"

Method Overloading: Compile-Time Polymorphism

Imagine you have a machine that processes payments. Sometimes you hand it cash, sometimes you hand it a credit card. It's doing the same conceptual action ("process payment"), but the internal mechanics are different depending on what you give it.

In Java, Method Overloading allows you to define multiple methods with the exact same name, as long as they require different parameters (different types, or a different number of parameters). The Java compiler looks at the arguments you pass in and automatically figures out which version of the method to execute.

JavaOverloading
public int add(int a, int b) { return a + b; }
public double add(double a, double b) { return a + b; }
public int add(int a, int b, int c) { return a + b + c; }

add(1, 2);       // calls add(int, int)
add(1.5, 2.5);   // calls add(double, double)
add(1, 2, 3);    // calls add(int, int, int)

// NOTE: Overloading is resolved at COMPILE TIME (not runtime)
// Return type alone does NOT differentiate overloaded methods

Varargs (Variable Arguments)

Accept any number of arguments of the same type using ... syntax.

JavaVarargs
// The vararg parameter must be the LAST parameter
public int sum(int... numbers) {
    int total = 0;
    for (int n : numbers) {
        total += n;
    }
    return total;
}

// Can call with any number of args
sum();               // 0
sum(1);              // 1
sum(1, 2, 3, 4, 5); // 15
sum(new int[]{1,2}); // Can also pass an array

// Mixing with other params
public void log(String prefix, Object... args) { }

Static vs Instance Methods

This is one of the most important distinctions in Java. Does the action belong to the blueprint, or does it belong to a specific house built from the blueprint?

  • Instance Methods: Require you to create an object first (new Calculator()). They can access and modify the specific state of that individual object.
  • Static Methods: Belong to the class itself. You do not need to create an object to use them (e.g., Math.max()). Because they don't belong to an object, they cannot access instance variables.
JavaStatic vs Instance
public class Calculator {
    private int memory = 0; // Instance field

    // Static method: belongs to the CLASS, not any instance
    // Call it: Calculator.add(1, 2)
    public static int add(int a, int b) {
        return a + b;
        // CANNOT access 'memory' here — no instance!
    }

    // Instance method: belongs to an OBJECT
    // Call it: calc.store(42)
    public void store(int value) {
        this.memory = value; // CAN access instance fields
    }
}

Pass-by-Value (Java's Golden Rule)

This is a common interview question and a frequent source of confusion. Java is STRICTLY pass-by-value. It is never pass-by-reference.

When you pass a variable into a method, Java makes a copy of that variable's value and hands the copy to the method.

  • For Primitives: The method gets a copy of the actual number. Changing the copy inside the method does nothing to the original.
  • For Objects (Reference Types): The variable holds a remote control (a memory address). Java copies the remote control and hands it to the method. If the method uses that copied remote to turn on the TV (modify the object), the TV turns on for everyone! But if the method throws away its remote and builds a new TV, your original remote still points to the old TV.
JavaPass-by-Value Demo
// Primitives: changes inside method do NOT affect the caller
public void tryChange(int x) { x = 99; }
int num = 10;
tryChange(num);
System.out.println(num); // Still 10!

// Objects: the reference is copied, so you CAN modify the object
public void addItem(List<String> list) { list.add("new"); }
List<String> myList = new ArrayList<>();
addItem(myList);
System.out.println(myList); // [new] — modified!

// But reassigning the reference does NOT affect the caller
public void tryReplace(List<String> list) {
    list = new ArrayList<>(); // Only local ref changes
}
tryReplace(myList);
System.out.println(myList); // Still [new] — unchanged!

Recursion: Inception

Recursion occurs when a method calls itself. It is a powerful technique for solving problems that can be broken down into smaller, identical sub-problems (like navigating a file directory tree or traversing a graph).

Every recursive method MUST have a base case—a condition that stops the recursion. Without a base case, the method will call itself infinitely, piling up memory on the Call Stack until the JVM crashes with a StackOverflowError.

JavaRecursion Examples
// Factorial: n! = n × (n-1) × ... × 1
public static int factorial(int n) {
    if (n <= 1) return 1;     // Base case
    return n * factorial(n - 1); // Recursive case
}

// Fibonacci
public static int fib(int n) {
    if (n <= 1) return n;
    return fib(n - 1) + fib(n - 2);
}

// Binary search (recursive)
public static int binarySearch(int[] arr, int target, int low, int high) {
    if (low > high) return -1;
    int mid = (low + high) / 2;
    if (arr[mid] == target) return mid;
    if (arr[mid] < target) return binarySearch(arr, target, mid + 1, high);
    return binarySearch(arr, target, low, mid - 1);
}
⚠️
StackOverflowErrorEvery recursive call adds a frame to the call stack. Too many recursive calls (no base case or very deep recursion) will crash with a StackOverflowError. Java does NOT support tail-call optimization.

Real-World Use Case: File System Crawlers

Imagine you are building a search feature for your computer's file system. You need to find a file named "secret.txt" inside a deeply nested folder structure.

This is a classic use case for Recursion. You write a method searchFolder(Folder f). If the folder contains the file, you return it. If it contains other folders, the method calls itself for each sub-folder. This allows you to traverse arbitrarily deep directory trees with very few lines of code.

Interview Preparation

Question 1
Overloading vs Overriding?
Overloading is compile-time polymorphism (same name, different parameters). Overriding is runtime polymorphism (subclass provides specific implementation of a parent's method).
Question 2
Can you overload a method based only on return type?
No. The compiler uses the method signature (name + parameters) to distinguish methods. Return type is ignored during overload resolution.
