Theory Deep Dive

External systems as unreliable boundaries that must be defended against

Mental Model

File systems and networks are fundamentally different from memory. They are slower by orders of magnitude, can fail partially, can be unavailable, can be contested by other processes, and have platform-specific behaviors. Good I/O code is defensive without being noisy.

Java has evolved its I/O model through three generations:

  • java.io (Java 1): Stream-oriented, blocking. Simple but scales poorly under high concurrency.
  • java.nio (Java 4): Buffer and channel-based. Non-blocking capable via Selectors. Enables one thread to serve thousands of connections.
  • java.nio.file (Java 7+): Modern file API (Files, Path, WatchService). Replaces the broken java.io.File.

Networking adds protocol concerns beyond connectivity: request format, response parsing, retry safety, idempotency, authentication, rate limiting, and backpressure. A successful TCP connection is only the beginning of a correct network operation.

Beginner-Friendly, Expert-Useful Lens

If you are new: Beginners should learn streams, readers, writers, sockets, URLs, and timeouts.

If you are revising: Experts should refresh buffering, character encodings, connection pooling, backpressure, retries, TLS, and partial failure.

Mastery check: You understand I/O and Networking when you can explain the simple mental model, write a tiny runnable example, identify the common trap, and describe how it behaves in a real project.

The Old java.io.File Problem

delete() returns boolean
Silent failure
Returns false when deletion fails. No exception, no reason, no way to diagnose.
mkdir() returns boolean
No info on failure
Returns false if directory already exists OR if creation fails. Indistinguishable.
list() returns null
NPE trap
Returns null (not empty array) when I/O error occurs. Guaranteed NullPointerException.
No charset control
Encoding bugs
FileReader uses platform default charset. Different behavior on Windows vs Linux.

JavaWriting files — with OpenOptions
import java.io.*;
import java.nio.charset.*;
import java.nio.file.*;
import java.util.*;

public class Main {
    public static void main(String[] args) throws IOException {
        Path logPath = Path.of("logs", "app.log");
        Files.createDirectories(logPath.getParent()); // create parent dirs if needed

        // Write (creates or replaces)
        Files.writeString(logPath, "Service started\n", StandardCharsets.UTF_8);

        // Append to existing file
        Files.writeString(logPath, "New event\n",
            StandardCharsets.UTF_8,
            StandardOpenOption.CREATE,
            StandardOpenOption.APPEND);

        // Write all lines
        Files.write(logPath, List.of("line1", "line2"), StandardCharsets.UTF_8);

        // Buffered writer for high-volume output
        List<String> events = List.of("event1", "event2");
        try (BufferedWriter bw = Files.newBufferedWriter(logPath, StandardCharsets.UTF_8,
                StandardOpenOption.CREATE, StandardOpenOption.APPEND)) {
            for (String e : events) {
                bw.write(e);
                bw.newLine();
            }
        }  // flush+close automatic
        System.out.println("Log written: " + logPath);
    }
}

Streams vs Channels — When to Use What

InputStream/OutputStream
Simple, Blocking
Best for text processing, small-medium files. Byte-at-a-time or buffered read. Always wrap in BufferedStream.
Channel + ByteBuffer
High Throughput
Block-level I/O. Supports non-blocking mode. Used in server frameworks (Netty) and high-speed file transfer.
Files.lines() Stream
Lazy + Memory-Safe
Preferred for processing large text files. Lazily loads lines, pipelines with Stream API. Must close the stream.
AsynchronousFileChannel
Non-Blocking Disk
I/O completed via CompletionHandler or Future. Useful in reactive/async pipelines for disk operations.
JavaHigh-performance file copy with FileChannel
import java.io.*;
import java.nio.*;
import java.nio.channels.*;
import java.nio.file.*;

public class Main {
    public static void main(String[] args) throws IOException {
        try (FileChannel src  = FileChannel.open(Path.of("large.bin"), StandardOpenOption.READ);
             FileChannel dest = FileChannel.open(Path.of("copy.bin"),
                     StandardOpenOption.CREATE, StandardOpenOption.WRITE)) {

            // Zero-copy transfer using OS-level sendfile()
            long transferred = src.transferTo(0, src.size(), dest);
            System.out.println("Transferred " + transferred + " bytes");
        }
        // transferTo() uses OS-level zero-copy (no JVM heap involved) — 5-10x faster than stream copy
    }
}

Modern HTTP Client (Java 11+)

JavaSynchronous and asynchronous requests
import java.net.*;
import java.net.http.*;
import java.time.*;
import java.net.URI;
import java.util.*;
import java.util.concurrent.*;
import static java.net.http.HttpResponse.BodyHandlers;

public class Main {
    public static void main(String[] args) throws Exception {
        // Build the client once and reuse (thread-safe, manages connection pools)
        HttpClient client = HttpClient.newBuilder()
            .version(HttpClient.Version.HTTP_2)
            .connectTimeout(Duration.ofSeconds(5))
            .build();

        // --- Synchronous GET ---
        HttpRequest getReq = HttpRequest.newBuilder()
            .uri(URI.create("https://httpbin.org/get"))
            .GET()
            .timeout(Duration.ofSeconds(10))
            .build();

        HttpResponse<String> resp = client.send(getReq, BodyHandlers.ofString());
        if (resp.statusCode() == 200) {
            String json = resp.body();
            System.out.println("Response: " + json.substring(0, Math.min(80, json.length())));
        }

        // --- POST with JSON body ---
        String json = """
            {"name": "Alice", "email": "alice@example.com"}
            """;

        HttpRequest postReq = HttpRequest.newBuilder()
            .uri(URI.create("https://httpbin.org/post"))
            .header("Content-Type", "application/json")
            .POST(HttpRequest.BodyPublishers.ofString(json))
            .build();

        HttpResponse<String> created = client.send(postReq, BodyHandlers.ofString());
        System.out.println("POST status: " + created.statusCode());

        // --- Asynchronous GET (non-blocking) ---
        CompletableFuture<HttpResponse<String>> future =
            client.sendAsync(getReq, BodyHandlers.ofString());

        future.thenApply(HttpResponse::body)
              .thenAccept(body -> System.out.println("Async body length: " + body.length()))
              .exceptionally(ex -> { System.out.println("Request failed: " + ex); return null; });

        future.join(); // wait for async to complete before program exits
    }
}
TIP
Always set .timeout() on the request AND .connectTimeout() on the client. Without timeouts, a hung remote server can block a thread indefinitely.

Reactive Streams & Backpressure

Java 11's HttpClient is actually built on the Reactive Streams specification (Java 9 Flow API). This means it natively supports Backpressure.

Imagine you are downloading a massive 5GB video file over a gigabit connection, but you are saving it to a slow, old spinning hard drive. In old Java, the network would pull data into memory much faster than the hard drive could write it, eventually causing an OutOfMemoryError. With the new HttpClient, the slow hard drive writer signals "backpressure" to the HttpClient, which automatically tells the remote server at the TCP layer to slow down the transmission rate, preventing memory explosion!

TCP Socket Programming

JavaClient socket with proper resource cleanup
import java.io.*;
import java.net.*;
import java.nio.charset.*;

public class Main {
    public static void main(String[] args) {
        try (Socket socket = new Socket()) {
            socket.connect(new InetSocketAddress("localhost", 8080), 3000); // 3s connect timeout
            socket.setSoTimeout(5000); // 5s read timeout

            try (PrintWriter out = new PrintWriter(
                        new BufferedWriter(new OutputStreamWriter(
                            socket.getOutputStream(), StandardCharsets.UTF_8)), true);
                 BufferedReader in = new BufferedReader(
                        new InputStreamReader(
                            socket.getInputStream(), StandardCharsets.UTF_8))) {

                out.println("PING");           // auto-flushed (true in PrintWriter constructor)
                String response = in.readLine();
                System.out.println("Got: " + response);
            }
        } catch (IOException e) {
            System.out.println("Could not connect: " + e.getMessage());
        }
        // socket auto-closed
    }
}

Non-Blocking I/O: Selectors (NIO)

In traditional blocking I/O, each connection needs a dedicated thread. At 10,000 connections you need 10,000 threads — ~80 GB RAM just for thread stacks. NIO Selectors let one thread multiplex thousands of connections:

JavaNon-blocking server with Selector
import java.io.*;
import java.net.*;
import java.nio.*;
import java.nio.channels.*;
import java.util.*;

public class Main {
    public static void main(String[] args) throws IOException {
        Selector selector = Selector.open();

        ServerSocketChannel serverChannel = ServerSocketChannel.open();
        serverChannel.bind(new InetSocketAddress("localhost", 8080));
        serverChannel.configureBlocking(false);                      // KEY: non-blocking mode
        serverChannel.register(selector, SelectionKey.OP_ACCEPT);

        ByteBuffer buffer = ByteBuffer.allocate(1024);

        // In production this loop runs forever; here we do 1 iteration for demo
        selector.selectNow();  // non-blocking select for demo

        Iterator<SelectionKey> keys = selector.selectedKeys().iterator();
        while (keys.hasNext()) {
            SelectionKey key = keys.next();
            keys.remove();                    // MUST remove manually!

            if (key.isAcceptable()) {
                SocketChannel client = serverChannel.accept();
                client.configureBlocking(false);
                client.register(selector, SelectionKey.OP_READ);

            } else if (key.isReadable()) {
                SocketChannel client = (SocketChannel) key.channel();
                buffer.clear();
                int bytesRead = client.read(buffer);

                if (bytesRead == -1) {
                    client.close();  // client disconnected
                } else {
                    buffer.flip();
                    client.write(buffer);  // echo back
                }
            }
        }
        selector.close();
        serverChannel.close();
        System.out.println("NIO selector demo complete.");
    }
}
INFO
In Java 21+, Virtual Threads are often a better alternative to NIO Selectors for most use cases. Virtual threads are cheap (thousands/millions fit in memory) and use blocking I/O with none of the callback complexity of NIO.

Production I/O Reliability Checklist

  • Always use try-with-resources — Never leave streams, channels, sockets, or connections open in error paths.
  • Always set timeouts — Connect timeout (when establishing connection) AND read timeout (when waiting for data).
  • Retry only idempotent operations — Retry GET requests. Never auto-retry a POST that creates a resource without idempotency keys.
  • Specify charset everywhere — StandardCharsets.UTF_8 on every Reader/Writer. Platform default is a footgun on Windows.
  • Use buffering — Wrap raw streams in BufferedReader/BufferedWriter. Raw byte-by-byte I/O can be 100x slower.
  • Log structured failure context — Include file path, operation, byte offset, response code. Bare "IOException" messages are useless in production.
  • Handle partial writes — Network sends are not guaranteed to write all bytes in one call. Loop on channel.write(buffer) until buffer is empty.

Principles That Matter

  • Always use java.nio.file.Files and Path in new code. Never use java.io.File.
  • Wrap streams in Buffered variants to minimize native I/O syscalls (huge performance gain).
  • Always specify charset explicitly: StandardCharsets.UTF_8. Never rely on platform default.
  • Close resources with try-with-resources. Every open stream is a leaked file handle until closed.
  • Set timeouts on all network calls. A hanging thread is worse than a failed request.

Common Misunderstandings

  • ⚠ Forgetting to flush — BufferedWriter may hold unflushed bytes. Try-with-resources calls close() which calls flush(). Manually flushing is needed for long-lived streams.
  • ⚠ Loading huge files into memory — Files.readAllLines() loads everything into a List<String>. Use Files.lines() (a lazy Stream<String>) for large files.
  • ⚠ Assuming UTF-8 everywhere — Windows systems may produce files in Windows-1252. Always detect or specify charset explicitly.
  • ⚠ Blocking HTTP in event loops — Blocking HTTP calls inside reactive chains or virtual thread pools can cause deadlocks. Use async variants.

Hard-Won Tips

  • ⚠ Every network call needs timeouts. A missing timeout can freeze a thread longer than any bug in your own code.
  • ⚠ Bytes are not text until decoded with the right charset. Platform defaults are a portability trap.
  • Do not assume one read call returns all data. Streams are chunked and can deliver partial content.
  • Connection reuse matters. Creating fresh connections for every request wastes latency and resources.
  • ⚠ Network success is not binary: slow responses, partial writes, stale DNS, TLS failures, rate limits, and retries shape real behavior.

Mastery Checklist

  • You use Files/Path for all file operations, never java.io.File.
  • You can stream large files line-by-line without memory explosion.
  • You can make synchronous and asynchronous HTTP requests with Java's HttpClient.
  • You can explain when NIO Selectors are better than thread-per-connection.
  • You can handle partial reads/writes and set connect/read timeouts.

Interview Preparation

Question 1
java.io vs java.nio — key differences?
java.io is stream-oriented and always blocking — one thread per connection. java.nio is buffer-and-channel oriented, supports non-blocking mode, and with Selectors allows one thread to handle thousands of connections. java.nio.file is the modern file API replacing java.io.File.
Question 2
What is Serialization and its security risks?
Converting an object graph to a byte stream. The class must implement Serializable. Risk: Java deserialization is a known attack vector — crafted byte streams can trigger arbitrary code execution. Prefer JSON (Jackson/Gson) or Protobuf for cross-system communication. Never deserialize untrusted data.
Question 3
How does Java's HttpClient differ from Apache HttpClient?
Java 11's HttpClient is built-in, supports HTTP/2, has an async/reactive API via CompletableFuture and BodySubscriber, and is immutable. Apache HttpClient is a third-party library, HTTP/1.1 by default, more configurable for edge cases like cookie handling and connection pools.
Question 4
Why use Files.lines() instead of Files.readAllLines()?
readAllLines() loads the entire file into a List<String> in memory — a 1 GB log file = 1 GB heap usage. Files.lines() returns a lazy Stream<String> that reads one line at a time. Critical: you MUST close the stream (try-with-resources) or the file handle leaks.
