Interfaces & Abstract Classes

Contracts, blueprints, and how to design flexible, extensible APIs.

TH

Theory Deep Dive

Interfaces as behavior contracts

Mental Model

An interface describes what an object can do without committing to how it does it. This separation lets high-level code depend on behavior and lets low-level implementations change independently.

Abstract classes are different: they can share partial implementation and state. Use an abstract class when subclasses truly share a base mechanism. Use an interface when callers only need a capability contract.

Default methods were added to let interfaces evolve without breaking all implementers. They are useful, but too many default methods can turn an interface into a hidden abstract class.

Principles That Matter

  • Interfaces define a contract. They specify WHAT a class should do, not HOW.
  • Program to an interface, not an implementation, for flexible architecture.
  • Java 8+ allows default and static methods in interfaces, breaking the "no implementation" rule for better API evolution.

Common Misunderstandings

  • Confusing Abstract Classes with Interfaces. Abstract classes can have state (instance variables); interfaces generally do not.
  • Thinking you must implement every method of an interface if you declare your class abstract. (You don't).
  • Overusing interfaces when a simple class or function would suffice.

Mastery Checklist

  • You can explain the difference between an Interface and an Abstract Class.
  • You know when and why to use 'default' methods.
  • You can explain multiple inheritance of type via interfaces.

Abstract Classes: The Unfinished Blueprint

Sometimes you want to create a parent class that is so generic, it doesn't make sense to ever build an actual object out of it. For example, a generic Vehicle. What does a "Vehicle" look like? You can build a Car or a Truck, but you can't just build a "Vehicle".

By declaring a class abstract, you prevent anyone from instantiating it with new. It becomes an unfinished blueprint. You can define abstract methods (methods with no body) that force the child classes to fill in the blanks, while also providing regular methods that all children will share.

Java
abstract class Vehicle {
    protected String brand;

    public Vehicle(String brand) { this.brand = brand; }

    // Abstract method — subclasses MUST implement
    public abstract void start();

    // Concrete method — shared by all subclasses
    public void stop() {
        System.out.println(brand + " stopped");
    }
}

class ElectricCar extends Vehicle {
    public ElectricCar(String brand) { super(brand); }

    @Override
    public void start() {
        System.out.println(brand + " silently starts");
    }
}

// Vehicle v = new Vehicle("X");  →  ❌ Cannot instantiate abstract class
Vehicle v = new ElectricCar("Tesla"); // ✅ Polymorphism

Interfaces: The Universal Contract

If an Abstract Class is about what an object is, an Interface is about what an object can do. Interfaces are the foundation of loose coupling and plug-and-play architecture in Java.

Think of a USB port. Your computer doesn't care if you plug in a mouse, a keyboard, or a fan. It only cares that the device strictly follows the USB contract (the physical shape and electrical pins). In Java, an Interface is that contract. Any class that implements an interface signs a legal contract promising to provide the methods defined in it.

JavaInterface Definition
interface Flyable {
    void fly();                      // Abstract (implicitly public abstract)
    int MAX_ALTITUDE = 10000;       // Constants (implicitly public static final)
}

interface Swimmable {
    void swim();
}

// A class can implement MULTIPLE interfaces
class Duck implements Flyable, Swimmable {
    @Override
    public void fly() { System.out.println("Duck flying"); }

    @Override
    public void swim() { System.out.println("Duck swimming"); }
}

// Use interface as type
Flyable f = new Duck();
f.fly();

Default Methods: Saving Backward Compatibility

Before Java 8, if you added a new method to a popular Interface, you would instantly break every single class in the world that implemented it (because they wouldn't have the new method implemented).

To fix this, Java introduced default methods. These allow you to add a method with a body directly inside the Interface. If a class implements the interface, it automatically inherits this default behavior without breaking, but can still choose to override it.

Java
interface Logger {
    // Abstract method
    void log(String msg);

    // Default method — provides an implementation, can be overridden
    default void info(String msg) {
        log("[INFO] " + msg);
    }

    // Static method — called on the interface itself
    static Logger getDefault() {
        return msg -> System.out.println(msg); // Lambda!
    }

    // Private method — helper for default methods (not visible outside)
    private String format(String msg) {
        return "[" + java.time.LocalTime.now() + "] " + msg;
    }
}

// Usage
Logger logger = Logger.getDefault();
logger.info("Hello");

Rule of ThumbUse interfaces to define what something can do (capabilities). Use abstract classes to define what something is (shared identity + code). A class can be both: abstract class Animal implements Flyable, Swimmable.

Sealed Classes (Modern Java)

In traditional Java, if you make a class public, anyone in the world can extend it. But sometimes you are building a strict domain model (like the states of a network request: Success, Error, Loading) and you want to mathematically guarantee that NO other states can ever exist.

Sealed classes allow you to definitively restrict which classes are allowed to extend your parent class. This gives the compiler superpowers—when you write a switch statement, the compiler knows exactly how many children exist, allowing for exhaustive pattern matching.

Java
// Only Circle, Rectangle, and Triangle can extend Shape
public sealed class Shape
    permits Circle, Rectangle, Triangle {
}

// Subclass options:
final class Circle extends Shape { }        // Cannot be extended further
sealed class Rectangle extends Shape
    permits Square { }                       // Can only be extended by Square
non-sealed class Triangle extends Shape { } // Open to any subclass

// Exhaustive switch (compiler checks all cases)
String desc = switch (shape) {
    case Circle c    -> "Circle";
    case Rectangle r -> "Rectangle";
    case Triangle t  -> "Triangle";
    // No default needed — compiler knows all subtypes!
};

Real-World Use Case: Plugin Architecture

Think about how web browsers support extensions. They define a strict contract (an interface). If you want to build an ad blocker, you must implement the AdBlockerInterface. The browser doesn't know who you are, but it knows you have the required buttons to press.

In Java applications, this is heavily used for database connections. The java.sql.Connection is an interface. Oracle, MySQL, and PostgreSQL all write classes that implement this interface. You can switch your database by changing a single line of configuration because your code only interacts with the interface, not the database-specific classes!

Interview Preparation

Question 1
What is a Marker Interface?
An interface with no methods or fields (e.g., Serializable, Cloneable). It is used to tag a class to signal special behavior to the JVM or a framework.
Question 2
Interface vs Abstract Class?
A class can implement multiple interfaces but extend only one abstract class. Interfaces cannot have state (instance variables), while abstract classes can.
