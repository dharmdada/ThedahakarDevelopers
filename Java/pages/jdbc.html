JDBC and SQL in Java

The low-level database API that every ORM is built on. Master it to understand what Hibernate is actually doing beneath the surface.

TH

Theory Deep Dive

Database access as boundary design

Mental Model

JDBC is the lowest layer in Java's database stack. It exposes the real responsibilities that ORMs hide: connection acquisition, statement compilation, parameter binding, result set iteration, transaction demarcation, and resource cleanup. Understanding JDBC explains why ORMs make the tradeoffs they do.

A database connection is an expensive resource — establishing one involves TCP handshake, authentication, and session negotiation. Connection pools (HikariCP, Tomcat JDBC) reuse connections by holding them open and lending them to application threads on demand.

SQL injection is not just bad practice — it is an existential security risk. A single concatenated string in a query can expose or delete an entire database. PreparedStatement prevents this by separating SQL structure from user data permanently.

Transactions exist because some business operations span multiple SQL statements. If a money transfer deducts from Account A but then fails before crediting Account B, the deducted money vanishes. Atomicity prevents this.

Beginner-Friendly, Expert-Useful Lens

If you are new: Beginners should learn Connection, PreparedStatement, ResultSet, transactions, and closing resources.

If you are revising: Experts should refresh isolation levels, pooling, batching, generated keys, SQL injection, mapping boundaries, and transaction ownership.

Mastery check: You understand JDBC when you can explain the simple mental model, write a tiny runnable example, identify the common trap, and describe how it behaves in a real project.

ℹ️
Core Java vs Third-Party LibrariesCore Java only provides the rules (Interfaces like Connection, PreparedStatement, and DataSource inside the java.sql package). Core Java does not know how to talk to PostgreSQL, MySQL, or Oracle. To actually connect to a database, you must download a third-party library called a JDBC Driver (e.g., the PostgreSQL JDBC Driver) and add it to your project. The Driver contains the actual classes (like PGSimpleDataSource) that implement the Core Java interfaces.

Principles That Matter

  • Always use PreparedStatement — Never concatenate user input into SQL. SQL injection is the #1 web vulnerability.
  • Close in reverse order — Close ResultSet first, then PreparedStatement, then Connection. Try-with-resources does this automatically.
  • Use DataSource, not DriverManager in production — DriverManager creates a new connection each call. DataSource pools them.
  • Wrap JDBC exceptions — SQLException is checked. Translate to domain exceptions at the repository boundary.

Common Misunderstandings

  • SQL Injection via concatenation — "WHERE name = '" + userInput + "'" is never acceptable. A user entering '; DROP TABLE users; -- destroys your database.
  • Not closing ResultSet — Each unclosed ResultSet is a cursor held open on the database server. Under load, this exhausts the DB's cursor limit.
  • N+1 query problem — Fetching 100 orders then looping to fetch each order's items with a separate query = 101 queries. Use JOINs or batch fetching.
  • Connection Leaks — A connection checked out from the pool but never closed (due to an unhandled exception or missing finally block) will never return to the pool. Over time, the pool empties, and the entire app freezes.
  • autoCommit=true for multi-step operations — Each statement commits individually. If step 2 fails, step 1 is already committed and cannot be rolled back.

Step 1: Making the Connection (From Scratch)

Before you can run any SQL, Java needs to connect to the database. Since Core Java only provides interfaces (empty rules), the very first thing you must do is add a Driver to your project (e.g., adding the PostgreSQL JDBC Driver to your `pom.xml` or `build.gradle`).

The Absolute Beginner Way: DriverManager

If you are just writing a quick script or learning JDBC for the very first time, you use the DriverManager. You hand it a database URL, a username, and a password, and it builds a physical network connection.

JavaHello World Database Connection
import java.sql.Connection;
import java.sql.DriverManager;
import java.sql.SQLException;

public class Main {
    public static void main(String[] args) {
        // The JDBC URL tells the DriverManager which database to find.
        // Format: jdbc:[driver-type]://[host]:[port]/[database_name]
        String url = "jdbc:postgresql://localhost:5432/my_database";
        String user = "postgres";
        String password = "supersecretpassword";

        // Try-with-resources ensures the connection is automatically closed when done!
        try (Connection con = DriverManager.getConnection(url, user, password);
             java.sql.Statement stmt = con.createStatement();
             java.sql.ResultSet rs = stmt.executeQuery("SELECT CURRENT_DATE")) {
             
            System.out.println("Successfully connected to the database!");
            if (rs.next()) {
                System.out.println("The database current date is: " + rs.getString(1));
            }
        } catch (SQLException e) {
            System.out.println("Failed to connect: " + e.getMessage());
        }
    }
}

The Production Way: DataSource

The DriverManager above has a fatal flaw: it is painfully slow. It takes 20 to 100 milliseconds to establish that TCP network connection. If 1,000 users visit your web app at the same time and run that code, your database will crash trying to open 1,000 new connections.

In the real world, we use a DataSource (specifically, a Connection Pool). A DataSource opens 10 connections the moment your app starts and keeps them alive in a "pool". When a user needs data, they borrow an already-open connection, use it, and instantly return it to the pool.

JavaConfiguring a DataSource
import org.postgresql.ds.PGSimpleDataSource;
import java.sql.Connection;
import java.sql.SQLException;

public class Main {
    public static void main(String[] args) throws SQLException {
        // 1. Configure the DataSource EXACTLY ONCE when the app starts
        PGSimpleDataSource ds = new PGSimpleDataSource();
        ds.setUrl("jdbc:postgresql://localhost:5432/my_database");
        ds.setUser("postgres");
        ds.setPassword("supersecretpassword");

        // 2. Pass the 'ds' object to your repositories. 
        // Now, getting a connection is instant (0.01ms instead of 50ms)!
        try (Connection con = ds.getConnection();
             java.sql.PreparedStatement ps = con.prepareStatement("SELECT 1 AS test_col");
             java.sql.ResultSet rs = ps.executeQuery()) {
             
            if (rs.next()) {
                System.out.println("Borrowed an instant connection! Database returned: " + rs.getInt("test_col"));
            }
        } 
        // When the try block ends, the connection is NOT destroyed. 
        // It is safely returned to the pool for the next user.
    }
}

Full CRUD Repository Pattern

JavaRepository class — clean, safe JDBC
import javax.sql.DataSource;
import java.sql.*;
import java.util.*;

record User(long id, String name, String email) {}

class UserRepository {
    private final DataSource ds;

    UserRepository(DataSource ds) {
        this.ds = ds;
    }

    Optional<User> findById(long id) {
        String sql = "SELECT id, name, email FROM users WHERE id = ?";
        try (Connection con = ds.getConnection();
             PreparedStatement ps = con.prepareStatement(sql)) {
            ps.setLong(1, id);
            try (ResultSet rs = ps.executeQuery()) {
                if (!rs.next()) return Optional.empty();
                return Optional.of(new User(
                    rs.getLong("id"),
                    rs.getString("name"),
                    rs.getString("email")
                ));
            }
        } catch (SQLException e) {
            throw new DataAccessException("Could not load user " + id, e);
        }
    }

    void save(User user) {
        String sql = "INSERT INTO users(id, name, email) VALUES (?, ?, ?)";
        try (Connection con = ds.getConnection();
             PreparedStatement ps = con.prepareStatement(sql)) {
            ps.setLong(1, user.id());
            ps.setString(2, user.name());
            ps.setString(3, user.email());
            ps.executeUpdate();
        } catch (SQLException e) {
            throw new DataAccessException("Could not save user", e);
        }
    }
}

class DataAccessException extends RuntimeException {
    DataAccessException(String message, Throwable cause) { super(message, cause); }
}

public class Main {
    public static void main(String[] args) {
        System.out.println("Create a DataSource, then pass it to new UserRepository(dataSource).");
    }
}

Pagination and Large Result Sets

Never return thousands of rows from a database in a single query. It consumes massive amounts of RAM and stalls the database. Always use Pagination to fetch data in chunks.

JavaLIMIT and OFFSET Pagination
import javax.sql.DataSource;
import java.sql.*;
import java.util.*;

record User(long id, String name, String email) {}

public class Main {
    static DataSource ds;

    public static void main(String[] args) {
        System.out.println("Inject a DataSource, then call getUsers(page, pageSize).");
    }

    public static List<User> getUsers(int page, int pageSize) throws SQLException {
        if (ds == null) return List.of();
        String sql = "SELECT id, name, email FROM users ORDER BY id LIMIT ? OFFSET ?";
        try (Connection con = ds.getConnection();
             PreparedStatement ps = con.prepareStatement(sql)) {
            ps.setInt(1, pageSize);
            ps.setInt(2, (page - 1) * pageSize);
            try (ResultSet rs = ps.executeQuery()) {
                List<User> users = new ArrayList<>();
                while (rs.next()) users.add(mapRow(rs));
                return users;
            }
        }
    }

    static User mapRow(ResultSet rs) throws SQLException {
        return new User(rs.getLong("id"), rs.getString("name"), rs.getString("email"));
    }
}
💡
Keyset Pagination (Seek Method): For massive tables, OFFSET gets incredibly slow because the database must scan and skip all previous rows. A faster approach is to filter by the last seen ID: WHERE id > ? ORDER BY id LIMIT ?.

Transaction Management

JavaAtomic money transfer — commit or full rollback
import javax.sql.DataSource;
import java.sql.*;
import java.math.BigDecimal;

public class Main {
    static DataSource ds;

    public static void main(String[] args) throws Exception {
        transfer(1001L, 1002L, new BigDecimal("250.00"));
    }

    public static void transfer(long fromId, long toId, BigDecimal amount) throws SQLException {
        try (Connection con = ds.getConnection()) {
            con.setAutoCommit(false);  // Start manual transaction

            try {
                // Debit
                try (PreparedStatement debit = con.prepareStatement(
                        "UPDATE accounts SET balance = balance - ? WHERE id = ? AND balance >= ?")) {
                    debit.setBigDecimal(1, amount);
                    debit.setLong(2, fromId);
                    debit.setBigDecimal(3, amount);       // optimistic check
                    int rows = debit.executeUpdate();
                    if (rows == 0) throw new IllegalStateException("Insufficient balance");
                }

                // Credit
                try (PreparedStatement credit = con.prepareStatement(
                        "UPDATE accounts SET balance = balance + ? WHERE id = ?")) {
                    credit.setBigDecimal(1, amount);
                    credit.setLong(2, toId);
                    credit.executeUpdate();
                }

                con.commit();  // ✅ Both succeeded — commit atomically

            } catch (SQLException | IllegalStateException e) {
                con.rollback(); // ❌ Any failure — undo everything
                throw new RuntimeException(e);
            }
            // Connection auto-closed by try-with-resources, returns to pool
        }
    }
}
WARN
Savepoints: JDBC supports partial rollbacks via Savepoint sp = con.setSavepoint() and con.rollback(sp). Use for nested logical operations where only the inner part should roll back.

Database Isolation Levels

When multiple transactions run at the same time, they can interfere with each other. The Isolation Level dictates how strict the database should be about preventing this interference. In JDBC, you can set this via Connection.setTransactionIsolation().

Isolation LevelDescriptionPerformance
READ_UNCOMMITTEDA transaction can see uncommitted changes from other transactions (Dirty Reads). Highly dangerous.Fastest
READ_COMMITTEDYou only see changes that have been committed. Default for Postgres and Oracle.Fast
REPEATABLE_READIf you read a row twice in a transaction, it will be the exact same, even if another transaction modified it in the meantime. Default for MySQL (InnoDB).Moderate
SERIALIZABLEThe strictest level. Transactions act as if they are running one after the other, completely eliminating concurrency issues like Phantom Reads.Slowest
JavaSetting Isolation Level
import javax.sql.DataSource;
import java.sql.*;

public class Main {
    static DataSource ds;

    public static void main(String[] args) throws SQLException {
        try (Connection con = ds.getConnection()) {
            // Prevent Dirty Reads and Non-Repeatable Reads for this specific transaction
            con.setTransactionIsolation(Connection.TRANSACTION_REPEATABLE_READ);
            con.setAutoCommit(false);

            // Execute critical queries with guaranteed isolation
            try (PreparedStatement ps = con.prepareStatement("SELECT balance FROM accounts WHERE id = ?")) {
                ps.setLong(1, 1001);
                try (ResultSet rs = ps.executeQuery()) {
                    if (rs.next()) {
                        System.out.println("Balance locked and read: " + rs.getBigDecimal("balance"));
                    }
                }
            }

            con.commit();
        }
    }
}

Batch Writes — 100x Faster Bulk Inserts

JavaBatching 10,000 rows efficiently
import javax.sql.DataSource;
import java.sql.*;
import java.util.*;

record Event(String type, String payload) {}

public class Main {
    static DataSource ds;

    public static void main(String[] args) throws SQLException {
        bulkInsert(List.of(new Event("LOGIN", "alice"), new Event("LOGOUT", "alice")));
    }

    public static void bulkInsert(List<Event> events) throws SQLException {
        if (ds == null) {
            System.out.println("Inject a DataSource before running against a database.");
            return;
        }
        String sql = "INSERT INTO event_log(type, payload) VALUES (?, ?)";
        try (Connection con = ds.getConnection();
             PreparedStatement ps = con.prepareStatement(sql)) {
            con.setAutoCommit(false);
            for (Event event : events) {
                ps.setString(1, event.type());
                ps.setString(2, event.payload());
                ps.addBatch();
            }
            ps.executeBatch();
            con.commit();
        }
    }
}

The N+1 Query Problem

Java❌ N+1: 1 query for orders + N queries for each order's items
import java.util.*;

record Order(long id) {}
record Item(long id, long orderId) {}
interface OrderRepository { List<Order> findAll(); }
interface ItemRepository { List<Item> findByOrderId(long orderId); }

public class Main {
    static OrderRepository orderRepo = () -> List.of(new Order(1), new Order(2));
    static ItemRepository itemRepo = orderId -> List.of(new Item(orderId * 10, orderId));

    public static void main(String[] args) {
        List<Order> orders = orderRepo.findAll();
        for (Order order : orders) {
            List<Item> items = itemRepo.findByOrderId(order.id());
            System.out.println(order + " -> " + items);
        }
    }
}
Java✅ Fixed: single JOIN query fetches everything at once
import javax.sql.DataSource;
import java.math.BigDecimal;
import java.sql.*;
import java.util.*;

record Item(long id, String productName, int quantity, BigDecimal price) {}
class Order {
    private final long id;
    private final long customerId;
    private final BigDecimal total;
    private final List<Item> items = new ArrayList<>();
    Order(long id, long customerId, BigDecimal total) {
        this.id = id;
        this.customerId = customerId;
        this.total = total;
    }
    void addItem(Item item) { items.add(item); }
    public String toString() { return "Order " + id + " items=" + items; }
}

public class Main {
    static DataSource ds;

    public static void main(String[] args) throws SQLException {
        System.out.println(fetchOrdersWithItems());
    }

    public static List<Order> fetchOrdersWithItems() throws SQLException {
        if (ds == null) return List.of();
        String sql = """
            SELECT o.id, o.customer_id, o.total,
                   i.id AS item_id, i.product_name, i.quantity, i.price
            FROM orders o
            LEFT JOIN order_items i ON i.order_id = o.id
            ORDER BY o.id
            """;
        try (Connection con = ds.getConnection();
             PreparedStatement ps = con.prepareStatement(sql);
             ResultSet rs = ps.executeQuery()) {
            Map<Long, Order> orderMap = new LinkedHashMap<>();
            while (rs.next()) {
                long orderId = rs.getLong("id");
                Order order = orderMap.computeIfAbsent(orderId,
                    id -> new Order(id, getLongQuietly(rs, "customer_id"), getBigDecimalQuietly(rs, "total")));
                long itemId = rs.getLong("item_id");
                if (!rs.wasNull()) {
                    order.addItem(new Item(itemId, rs.getString("product_name"), rs.getInt("quantity"), rs.getBigDecimal("price")));
                }
            }
            return new ArrayList<>(orderMap.values());
        }
    }

    private static long getLongQuietly(ResultSet rs, String column) {
        try { return rs.getLong(column); } catch (SQLException e) { throw new RuntimeException(e); }
    }

    private static BigDecimal getBigDecimalQuietly(ResultSet rs, String column) {
        try { return rs.getBigDecimal(column); } catch (SQLException e) { throw new RuntimeException(e); }
    }
}

Connection Pooling with HikariCP

JavaHikariCP configuration — production settings
import com.zaxxer.hikari.HikariConfig;
import com.zaxxer.hikari.HikariDataSource;
import javax.sql.DataSource;

public class Main {
    public static void main(String[] args) {
        HikariConfig config = new HikariConfig();
        config.setJdbcUrl("jdbc:postgresql://db-host:5432/myapp");
        config.setUsername("app_user");
        config.setPassword(System.getenv("DB_PASSWORD"));  // never hardcode!

        // Pool sizing: rule of thumb = (num_cpu_cores * 2) + num_disk_spindles
        config.setMaximumPoolSize(10);
        config.setMinimumIdle(5);

        // Connection timeouts
        config.setConnectionTimeout(3000);   // fail fast if no connection available (ms)
        config.setIdleTimeout(600_000);      // remove idle connections after 10 min
        config.setMaxLifetime(1_800_000);    // force recycle connections after 30 min

        // Validation
        config.setConnectionTestQuery("SELECT 1");  // for drivers lacking isValid()
        config.setKeepaliveTime(60_000);            // heartbeat to prevent firewall drops

        // Performance
        config.addDataSourceProperty("cachePrepStmts",         "true");
        config.addDataSourceProperty("prepStmtCacheSize",      "250");
        config.addDataSourceProperty("prepStmtCacheSqlLimit",  "2048");
        config.addDataSourceProperty("useServerPrepStmts",     "true"); // MySQL-specific

        DataSource ds = new HikariDataSource(config);
        // Thread-safe singleton — create once, share everywhere
        System.out.println("HikariCP DataSource ready: " + ds);
    }
}
TIP
The optimal pool size is almost never "as many as possible". Too many connections overwhelm the database. For a 4-core PostgreSQL server, start with 10 connections max and benchmark before increasing.

Interview Preparation

Question 1
Statement vs PreparedStatement?
Statement compiles SQL each time and concatenates values — vulnerable to injection. PreparedStatement compiles once, binds parameters safely, is reusable, and is 2-5x faster for repeated execution. Always use PreparedStatement.
Question 2
What is SQL Injection and how do you prevent it?
Injecting SQL commands via user input (e.g., '; DROP TABLE users--). Prevention: ALWAYS use PreparedStatement with parameter binding, never concatenate user strings into SQL. Input validation is secondary — PreparedStatement is the primary defense.
Question 3
What is the N+1 problem?
Fetching a list of N entities then making N additional queries for each entity's associated data. Fix: use a JOIN to load everything in one query and build the object graph in Java, or use batched IN-clause queries.
Question 4
Why use a connection pool?
Creating a DB connection costs 20-100ms (TCP + auth). Pooling reuses open connections, reducing request latency to microseconds. HikariCP is the fastest JVM pool — used by default in Spring Boot.
