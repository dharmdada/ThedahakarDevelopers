JDBC and SQL in Java

Low-level database programming: prepared statements, transactions, batching, and connection management.

TH

Theory Deep Dive

Database access as boundary design

Mental Model

JDBC is the low-level contract between Java and relational databases. It exposes the real responsibilities that ORMs often hide: connections, statements, result sets, transactions, parameter binding, and resource cleanup.

PreparedStatement is essential because SQL text and user values must be separated. This is a correctness issue and a security issue. Parameter binding prevents user input from changing the structure of the SQL command.

Transactions define consistency boundaries. If several writes represent one business action, they must commit or roll back together. Correct transaction handling is more important than simply making queries run.

Safe Query Pattern

JavaPrepared Statement with try-with-resources
String sql = "SELECT id, name, email FROM users WHERE status = ?";

try (Connection con = dataSource.getConnection();
     PreparedStatement ps = con.prepareStatement(sql)) {

    ps.setString(1, "ACTIVE");

    try (ResultSet rs = ps.executeQuery()) {
        while (rs.next()) {
            long id = rs.getLong("id");
            String name = rs.getString("name");
            String email = rs.getString("email");
            // map row to domain object
        }
    }
}
WARN
Never build SQL by string concatenation with user input. Always bind parameters through PreparedStatement.

Transaction Handling

Use transactions whenever multiple statements must succeed or fail as one unit.

Java
Connection con = dataSource.getConnection();
try {
    con.setAutoCommit(false);

    try (PreparedStatement debit = con.prepareStatement(
             "UPDATE account SET balance = balance - ? WHERE id = ?");
         PreparedStatement credit = con.prepareStatement(
             "UPDATE account SET balance = balance + ? WHERE id = ?")) {

        debit.setBigDecimal(1, amount);
        debit.setLong(2, fromAccountId);
        debit.executeUpdate();

        credit.setBigDecimal(1, amount);
        credit.setLong(2, toAccountId);
        credit.executeUpdate();
    }

    con.commit();
} catch (Exception e) {
    con.rollback();
    throw e;
} finally {
    con.close();
}

Batch Writes for Performance

For large inserts/updates, batch operations reduce network round trips.

Java
String insert = "INSERT INTO event_log(type, payload) VALUES(?, ?)";
try (Connection con = dataSource.getConnection();
     PreparedStatement ps = con.prepareStatement(insert)) {
    con.setAutoCommit(false);
    for (Event e : events) {
        ps.setString(1, e.type());
        ps.setString(2, e.payload());
        ps.addBatch();
    }
    ps.executeBatch();
    con.commit();
}

Production Best Practices

  • Use connection pooling (HikariCP or app server pool).
  • Set query timeouts to avoid hanging threads.
  • Index columns used in joins and frequent filters.
  • Keep SQL in repository classes, not controller classes.
  • Map checked SQL exceptions to domain-specific runtime exceptions at boundaries.
