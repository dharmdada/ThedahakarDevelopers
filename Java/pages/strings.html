Strings

Everything about Java's most used class. Strings are immutable objects.

TH

Theory Deep Dive

Text as immutable value data

Mental Model

String is immutable: once created, its character content cannot change. Operations like replace, substring, and toUpperCase return new strings. This design makes strings safe to share, cache, intern, and use as map keys.

Because strings are objects, equality must be understood carefully. The == operator checks whether two references point to the same object. The equals method checks text content. This distinction is one of the first object identity lessons in Java.

StringBuilder exists because repeated concatenation in loops can create unnecessary temporary objects. Use String for stable text, StringBuilder for incremental construction, and formatter APIs when structure matters.

Principles That Matter

  • Strings are strictly immutable. Once created, they cannot be changed.
  • Understand the String Pool. Literal strings are cached and reused to save memory.
  • Use StringBuilder for string manipulation in loops to avoid memory bloat.

Common Misunderstandings

  • Using == instead of .equals() for string comparison. This is the #1 beginner mistake.
  • Thinking String methods like .trim() or .replace() modify the original string. They return a NEW string.
  • Concatenating strings inside a loop using the '+' operator, which creates many hidden objects.

Mastery Checklist

  • You can explain why Strings are immutable and the security/performance benefits.
  • You know the difference between String, StringBuilder, and StringBuffer.
  • You can write a regex to split or validate a complex string.

The String Pool: The Public Library

In Java, Strings are Immutable. Once a String is created, it can never be changed. If you try to modify a String, Java actually creates a brand new String and throws the old one away.

Because Strings are used so heavily, Java uses an optimization called the String Pool. Think of it like a public library. If you create the string "Hello", Java puts that book in the library. If another variable asks for "Hello", Java doesn't print a new book; it just gives them the exact same book from the library to save RAM.

Java
// String literal — stored in String Pool (shared)
String s1 = "Hello";

// Using new — creates a new object on the heap (avoid this)
String s2 = new String("Hello");

// From char array
char[] chars = {'H', 'i'};
String s3 = new String(chars); // "Hi"

// Concatenation
String full = "Hello" + " " + "World";
String joined = String.join(", ", "a", "b", "c"); // "a, b, c"

StringBuilder: The Scratchpad

Because Strings are immutable, concatenating Strings inside a loop is a massive performance killer. Every time you write str += "a", Java is forced to build a brand new String object, copy all the old data into it, add the "a", and throw the old one in the trash. Doing this 1,000 times creates 1,000 abandoned objects in your Heap.

StringBuilder solves this. It acts like a mutable scratchpad. You can write, erase, and append to it infinitely without creating new objects. Only when you are completely finished do you call .toString() to mint the final, immutable String.

Java
// ❌ Bad: creates 1000 temporary String objects
String s = "";
for (int i = 0; i < 1000; i++) {
    s += i; // Creates new String each time!
}

// ✅ Good: uses a mutable buffer
StringBuilder sb = new StringBuilder();
for (int i = 0; i < 1000; i++) {
    sb.append(i);
}
String result = sb.toString();

// StringBuilder methods: append, insert, delete, reverse, replace
sb.insert(0, "Start: ");
sb.reverse();
sb.delete(0, 5);
ℹ️
StringBuilder vs StringBufferStringBuilder is NOT thread-safe (faster). StringBuffer IS thread-safe (slower). Use StringBuilder unless multiple threads are modifying the same string.

String Comparison: Identity vs Value

This is the most common bug for Java beginners. Never use == to compare Strings.

  • == (Identity): Asks the question: "Are these two remote controls pointing to the exact same physical object in memory?"
  • .equals() (Value): Asks the question: "Do these two objects contain the exact same characters?"
Java
String a = "hello";
String b = "hello";
String c = new String("hello");

// == compares REFERENCES (memory address)
a == b   // true (same string pool object)
a == c   // FALSE! (different objects in memory!)

// .equals() compares VALUES (correct way!)
a.equals(c)             // true
a.equalsIgnoreCase("HELLO") // true

// compareTo returns: negative, 0, or positive
"apple".compareTo("banana")  // negative (a < b)
"banana".compareTo("apple")  // positive (b > a)
"apple".compareTo("apple")   // 0 (equal)

Regular Expressions (Basics)

Java
import java.util.regex.*;

// Simple matching
boolean match = "hello123".matches("[a-z]+\\d+"); // true

// Pattern + Matcher (for complex use)
Pattern p = Pattern.compile("\\d+"); // Match digits
Matcher m = p.matcher("abc 123 def 456");
while (m.find()) {
    System.out.println(m.group()); // "123", then "456"
}

// Common patterns:
// \\d  = digit       \\D  = non-digit
// \\w  = word char   \\W  = non-word
// \\s  = whitespace  \\S  = non-whitespace
// .    = any char    +    = one or more
// *    = zero or more  ?  = zero or one
// [abc]= any of a,b,c  [^abc]= NOT a,b,c

Real-World Use Case: Log Parsers

Imagine you are building a log analysis tool for a server. The server generates millions of lines of text logs every day. You need to scan these logs to find error codes and IP addresses.

By using Regular Expressions and StringBuilder, you can parse these massive files efficiently. You use Regex to find patterns (like IP addresses) and use StringBuilder to aggregate report summaries without crashing the server's memory with millions of temporary String objects.

Interview Preparation

Question 1
What is the String Pool?
A special storage area in the Java Heap. When a string literal is created, the JVM checks the pool. If it exists, it returns the reference, saving memory.
Question 2
StringBuilder vs StringBuffer?
Both are mutable sequences of characters. StringBuffer is thread-safe (synchronized), whereas StringBuilder is not, making StringBuilder faster for single-threaded use.
