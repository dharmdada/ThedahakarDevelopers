Reflection & Annotations

Inspect and modify code at runtime, and attach metadata to your classes.

TH

Theory Deep Dive

Runtime metadata and controlled dynamism

Mental Model

Reflection lets Java inspect and interact with classes, methods, fields, constructors, and annotations at runtime. This is how many frameworks discover components, inject dependencies, map JSON, run tests, and connect annotations to behavior.

Reflection is powerful because it bypasses normal compile-time relationships. That same power makes it slower, harder to refactor, and easier to misuse. Use it for framework-like infrastructure, not ordinary business logic.

Annotations are metadata, not behavior by themselves. Something must read the annotation and decide what it means. Understanding that separation makes frameworks feel less magical.

Beginner-Friendly, Expert-Useful Lens

If you are new: Beginners should understand that reflection can access classes, fields, methods, and annotations at runtime.

If you are revising: Experts should refresh access checks, performance, modules, proxies, annotation processing alternatives, and framework internals.

Mastery check: You understand Reflection when you can explain the simple mental model, write a tiny runnable example, identify the common trap, and describe how it behaves in a real project.

Principles That Matter

  • Reflection allows inspecting and modifying code at runtime (e.g., reading private fields).
  • Use it sparingly. It breaks encapsulation and is slower than direct code execution.
  • Frameworks (Spring, Hibernate, JUnit) rely heavily on reflection to work their magic (dependency injection, mapping, testing).

Common Misunderstandings

  • Using reflection to bypass security restrictions in normal application logic.
  • Thinking reflection is fast. It involves dynamic resolution, preventing JIT optimizations.
  • Forgetting that reflection makes code brittle to refactoring (e.g., renaming a field breaks reflection code looking for it by name).

Mastery Checklist

  • You can get the Class object of a type.
  • You can read a private field's value using reflection.
  • You understand how annotations and reflection work together to power frameworks.

Hard-Won Tips

  • Reflection is why many frameworks feel magical. Understanding it makes dependency injection, ORM mapping, and serialization much less mysterious.
  • Reflective code bypasses many compile-time guarantees, so failures move to runtime and need stronger tests.
  • setAccessible can break encapsulation and may be restricted by modules or security policies.
  • Reflection is often acceptable at startup but risky in hot loops unless cached and measured.
  • If you control both sides, code generation or explicit interfaces may be cleaner than reflective discovery.

Reflection: X-Ray Vision for Java

Normally, a Java program is like a machine: it runs the code you wrote, but it doesn't know what it's running. It can't look at itself.

Reflection gives Java X-Ray vision. It allows a running program to inspect its own code. At runtime, a program can say, "Show me every method inside the `User` class," or "What are the names of all the private variables here?" This is the magic that allows frameworks like Spring to automatically find and wire your classes together without you writing explicit setup code.

JavaInspecting a Class
import java.lang.reflect.*;

public class Main {
    public static void main(String[] args) throws Exception {
        Class<?> clazz = String.class;

        System.out.println("Name: " + clazz.getName());
        System.out.println("Is interface? " + clazz.isInterface());

        Method[] methods = clazz.getMethods();
        for (int i = 0; i < Math.min(5, methods.length); i++) {
            System.out.println("Method: " + methods[i].getName());
        }

        Field[] fields = clazz.getDeclaredFields();
        System.out.println("Declared field count: " + fields.length);
    }
}

Dynamic Interaction: The Lockpick

Reflection isn't just for looking—it's for touching. Have you ever been told that a private variable is strictly hidden and can never be accessed from the outside? That is a lie.

Using Reflection, you can act like a lockpick. You can dynamically find a private variable, force it to become accessible using setAccessible(true), and change its value. This breaks all rules of Encapsulation, which is why it is considered highly dangerous in normal application code.

JavaModifying Private Fields
import java.lang.reflect.Field;

class Secret {
    private String password = "12345";
}

public class Main {
    public static void main(String[] args) throws Exception {
        Secret obj = new Secret();
        Class<?> clazz = obj.getClass();

        // Access private field
        Field field = clazz.getDeclaredField("password");
        field.setAccessible(true); // BYPASS PRIVATE MODIFIER!

        // Read value
        String val = (String) field.get(obj);
        System.out.println("Old: " + val);

        // Change value
        field.set(obj, "hacked!");
    }
}
⚠️
Reflection is Slow and DangerousReflection bypasses compile-time safety and is significantly slower than direct access. Use it only when writing frameworks or highly generic utility code, never for regular application logic.

Annotations: Sticky Notes for the Compiler

An Annotation (@) is literally just a sticky note you attach to your code. If you put an @Deprecated sticky note on a method, it doesn't magically change how the method executes. It simply sits there.

The magic happens when other programs read those sticky notes. The Java Compiler reads the @Override sticky note and verifies your syntax. Spring reads the @RestController sticky note and automatically sets up a web server for that class. They are pure metadata.

JavaBuilt-in Annotations
@FunctionalInterface
interface Converter {
    String convert(int value);
}

class Parent {
    void run() { System.out.println("Parent"); }
}

class Child extends Parent {
    @Override
    void run() { System.out.println("Child"); }

    @Deprecated(since = "1.5", forRemoval = false)
    void oldApi() { System.out.println("Deprecated API"); }

    @SuppressWarnings("unchecked")
    void uncheckedExample() {
        java.util.List raw = new java.util.ArrayList();
        java.util.List<String> typed = raw;
    }
}

public class Main {
    public static void main(String[] args) {
        new Child().run();
        Converter converter = value -> "#" + value;
        System.out.println(converter.convert(42));
    }
}

Creating Custom Annotations

JavaCustom Annotation
import java.lang.annotation.*;

@Retention(RetentionPolicy.RUNTIME)
@Target({ElementType.METHOD, ElementType.TYPE})
@interface TestInfo {
    String author() default "unknown";
    int priority() default 1;
    String[] tags() default {};
}

@TestInfo(author = "Alice", priority = 5, tags = {"fast", "db"})
class MyTests {
    @TestInfo
    public void test1() {}
}

public class Main {
    public static void main(String[] args) throws Exception {
        Class<?> clazz = MyTests.class;
        if (clazz.isAnnotationPresent(TestInfo.class)) {
            TestInfo info = clazz.getAnnotation(TestInfo.class);
            System.out.println("Author: " + info.author());
        }

        TestInfo methodInfo = clazz.getMethod("test1").getAnnotation(TestInfo.class);
        System.out.println("Method priority: " + methodInfo.priority());
    }
}

Real-World Use Case: Frameworks like Spring and JUnit

Reflection is the magic engine behind almost all modern Java frameworks.

When you use **Spring Boot** and add the @Autowired annotation, Spring uses reflection to scan your classes, find private fields, and inject dependencies directly into them—even if they have no setter methods! Similarly, **JUnit** uses reflection to scan your test classes, find all methods marked with @Test, and execute them automatically. Without reflection, you would have to write thousands of lines of manual configuration code.

Interview Preparation

Question 1
What is Reflection?
An API that allows a Java program to examine or modify the runtime behavior of applications running in the JVM. It can inspect classes, interfaces, fields, and methods at runtime.
Question 2
Why is Reflection slow?
Because it involves types that are dynamically resolved, certain JVM optimizations cannot be performed. It also involves type checking and security checks at runtime instead of compile-time.
