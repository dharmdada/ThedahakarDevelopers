Inheritance & Polymorphism

Reuse code through class hierarchies and write flexible, extensible programs.

TH

Theory Deep Dive

Inheritance as substitutable specialization

Mental Model

Inheritance means an object of the child type can be used where the parent type is expected. That substitution promise is more important than code reuse. If a subclass breaks expectations of the parent, the design is wrong even if it compiles.

Overriding enables runtime polymorphism: Java chooses the implementation based on the actual object, not the variable type. This is powerful because callers can depend on a stable parent contract while implementations vary.

The Object methods equals, hashCode, and toString influence how objects behave in collections, logs, debugging, and comparisons. Poor implementations create bugs far from the class where the mistake was made.

Beginner-Friendly, Expert-Useful Lens

If you are new: Beginners should learn extends, overriding, super, and why a child object can be used through a parent reference.

If you are revising: Experts should refresh Liskov substitution, fragile base classes, final methods, protected leakage, and composition alternatives.

Mastery check: You understand Inheritance when you can explain the simple mental model, write a tiny runnable example, identify the common trap, and describe how it behaves in a real project.

Principles That Matter

  • Inheritance models an "IS-A" relationship. A Dog IS-A Animal.
  • Prefer Composition over Inheritance when the relationship is "HAS-A".
  • The 'super' keyword accesses the immediate parent class's methods and constructors.

Common Misunderstandings

  • Using inheritance just to reuse code (code reuse is better achieved via composition).
  • Forgetting that private members of a parent class are NOT accessible in the child class directly.
  • Misunderstanding method hiding (for static methods) vs method overriding (for instance methods).

Mastery Checklist

  • You can explain polymorphism and how method overriding enables it.
  • You know the role of the Object class as the root of all Java classes.
  • You can explain what 'protected' access means.

Advanced Notes Most Courses Skip

Inheritance is a strict substitution promise, not a shortcut for sharing code.

Beginner and Expert Lens

Beginner lens: Beginners should learn extends, overriding, super, and why a child object can be used through a parent reference.

Expert refresh: Experts should refresh Liskov substitution, fragile base classes, final methods, protected leakage, and composition alternatives.

Hard-Won Tips

  • If a subclass weakens a parent promise, the hierarchy is broken even if the compiler accepts it.
  • Protected fields are often worse than public methods because they expose representation to every subclass forever.
  • Base classes become hard to change because subclasses may depend on undocumented call order or internal hooks.
  • Prefer inheritance for true is-a behavior with stable contracts; prefer composition for reuse and changing behavior.
  • Making a class final can be a design strength when extension would break invariants.

Inheritance: The "IS-A" Relationship

Inheritance allows a new class (the Child or Subclass) to inherit all the non-private fields and methods from an existing class (the Parent or Superclass). We use inheritance to model an IS-A relationship (e.g., a Dog IS-A Animal).

Unlike languages like C++, Java strictly enforces single inheritance for classes. A class can only extend one parent. This prevents the infamous "Deadly Diamond of Death" (where a child inherits conflicting methods from two parents).

JavaBasic Inheritance
public class Main {
    public static void main(String[] args) {
        Dog d = new Dog("Rex", "Lab");
        d.eat();  // Inherited from Animal
        d.bark(); // Own method
    }
}

class Animal {
    protected String name;

    public Animal(String name) { this.name = name; }

    public void eat() {
        System.out.println(name + " is eating");
    }
}

class Dog extends Animal {
    private String breed;

    public Dog(String name, String breed) {
        super(name);           // MUST call parent constructor first
        this.breed = breed;
    }

    public void bark() { System.out.println("Woof!"); }
}

The Diamond Problem

Java forbids multiple inheritance for classes. If a class could extend both Machine and Animal, and both had a move() method, which one would the child inherit? This ambiguity is called the Diamond Problem. By restricting classes to extends only one parent, Java eliminates this bug entirely.

The super Keyword: Calling the Parent

Just as this refers to "myself", the super keyword refers to "my parent". When a child class is created, the JVM must first fully create the parent portion of that object. Therefore, the very first line of any child constructor must be a call to super() to initialize the parent.

Java
public class Main {
    public static void main(String[] args) {
        Child c = new Child();
        c.demo();
    }
}

class Parent {
    protected int x = 10;
    public void show() { System.out.println("Parent"); }
}

class Child extends Parent {
    private int x = 20; // Hides parent's x

    public void demo() {
        System.out.println(x);        // 20 (own field)
        System.out.println(super.x);  // 10 (parent's field)
        super.show();                // Call parent's method
    }
}

Method Overriding

A subclass provides its own implementation of a method defined in its parent. The method signature must match exactly.

Java
public class Main {
    public static void main(String[] args) {
        Shape s1 = new Circle(5.0);
        Shape s2 = new Rectangle(4.0, 6.0);
        System.out.println("Circle area: " + s1.area());
        System.out.println("Rectangle area: " + s2.area());
    }
}

class Shape {
    public double area() { return 0; }
}

class Circle extends Shape {
    private double radius;
    public Circle(double r) { this.radius = r; }

    @Override  // Always use this annotation — catches typos at compile time
    public double area() {
        return Math.PI * radius * radius;
    }
}

class Rectangle extends Shape {
    private double w, h;
    public Rectangle(double w, double h) { this.w = w; this.h = h; }

    @Override
    public double area() { return w * h; }
}
ℹ️
Overriding Rules • Same method name and parameters
• Return type must be same or a subtype (covariant return)
• Access modifier can be same or MORE visible (not less)
• Cannot override final, static, or private methods

Polymorphism: Many Forms

Polymorphism is the true superpower of Object-Oriented Programming. The word literally translates to "many forms". In Java, it means that a variable of a Parent type can safely hold a reference to an object of ANY Child type.

If you have an Animal remote control, you can point it at a Dog TV, a Cat TV, or a Bird TV. When you press the speak() button on the remote, Java dynamically figures out which TV it is pointing to at runtime, and executes the correct overridden method (bark, meow, or chirp).

JavaRuntime Polymorphism
public class Main {
    public static void main(String[] args) {
        // Parent reference → child object
        Shape s1 = new Circle(5);
        Shape s2 = new Rectangle(4, 6);

        // Java calls the OVERRIDDEN method based on actual object type
        System.out.println(s1.area()); // Circle's area: 78.54
        System.out.println(s2.area()); // Rectangle's area: 24.0

        Main demo = new Main();
        demo.printArea(new Circle(3));
        demo.printArea(new Rectangle(2, 5));
    }

    // Power of polymorphism: one method handles ALL shapes
    public void printArea(Shape shape) {
        System.out.println("Area: " + shape.area());
    }
}

class Shape {
    public double area() { return 0; }
}

class Circle extends Shape {
    private double radius;
    public Circle(double r) { this.radius = r; }

    @Override
    public double area() {
        return Math.PI * radius * radius;
    }
}

class Rectangle extends Shape {
    private double w, h;
    public Rectangle(double w, double h) { this.w = w; this.h = h; }

    @Override
    public double area() { return w * h; }
}
💡
Liskov Substitution Principle (LSP)
A core SOLID principle: Objects of a superclass shall be replaceable with objects of its subclasses without breaking the application. If your subclass throws an UnsupportedOperationException for a method it inherited, you are violating LSP.

Upcasting & Downcasting: Changing Perspectives

Casting objects doesn't physically change the object in memory. It only changes the type of the remote control you are using to interact with the object.

  • Upcasting (Safe): Treating a Dog as an Animal. You lose access to dog-specific methods (like fetch()) because the Animal remote control doesn't have a fetch button. This is done automatically by Java.
  • Downcasting (Risky): You have an Animal remote control, and you want to swap it for a Dog remote control to press the fetch() button. If the object isn't actually a Dog, the JVM will throw a massive ClassCastException crash. You must explicitly cast it.
Java
public class Main {
    public static void main(String[] args) {
        // Upcasting: child → parent (always safe, automatic)
        Dog dog = new Dog("Rex", "Lab");
        Animal animal = dog; // Automatic upcasting
        animal.eat();     // ✅ Works
        // animal.bark();  →  ❌ Compile error! Animal doesn't have bark()

        // Downcasting: parent → child (risky, manual)
        if (animal instanceof Dog d) {  // Pattern matching — safe check + cast
            d.bark(); // ✅ Safe!
        }

        // Without pattern matching (older style)
        if (animal instanceof Dog) {
            Dog d2 = (Dog) animal; // Manual cast
            d2.bark();
        }

        // Bad downcast → ClassCastException at runtime!
        // Cat cat = (Cat) animal;  →  💥 animal is actually a Dog!
    }
}

class Animal {
    protected String name;
    public Animal(String name) { this.name = name; }
    public void eat() { System.out.println(name + " is eating"); }
}

class Dog extends Animal {
    private String breed;
    public Dog(String name, String breed) {
        super(name);
        this.breed = breed;
    }
    public void bark() { System.out.println("Woof!"); }
}

The Object Class: The Ancestor of Everything

In Java, the java.lang.Object class is the absolute root of the inheritance tree. Even if you don't write extends, your class secretly extends Object. This guarantees that every single object in Java shares a common set of foundational methods.

Java
import java.util.Objects;

public class Main {
    public static void main(String[] args) {
        Person p1 = new Person("Alice", 25);
        Person p2 = new Person("Alice", 25);
        System.out.println(p1);
        System.out.println(p1.equals(p2)); // true
    }
}

class Person {
    private String name;
    private int age;

    public Person(String name, int age) {
        this.name = name;
        this.age = age;
    }

    // toString — called by println, string concat, debugging
    @Override
    public String toString() {
        return "Person{name='%s', age=%d}".formatted(name, age);
    }

    // equals — defines logical equality
    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Person p)) return false;
        return age == p.age && Objects.equals(name, p.name);
    }

    // hashCode — MUST override if you override equals
    @Override
    public int hashCode() {
        return Objects.hash(name, age);
    }
}
🔥
equals/hashCode ContractIf two objects are equal(), they MUST have the same hashCode(). Violating this breaks HashMap, HashSet, and other hash-based collections.

Real-World Use Case: UI Frameworks

Look at Android Development or JavaFX. Every single button, text field, and image on the screen is a subclass of a parent View class.

The OS rendering engine doesn't care if it's a Button or a TextField. It just loops through a List<View> and calls view.draw(). Thanks to Polymorphism, the Button draws itself as a rectangle with text, and the Image draws pixels. This is the exact reason UI engines can render thousands of completely different components using just one loop.

Interview Preparation

Question 1
Does Java support multiple inheritance?
Java does not support multiple inheritance for classes to avoid the "Diamond Problem". However, a class can implement multiple interfaces.
Question 2
What is Polymorphism?
The ability of a single interface or reference to take on multiple forms. In Java, this usually means a parent reference variable can point to any child object.
