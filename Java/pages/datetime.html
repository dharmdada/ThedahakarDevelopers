Date and Time API

Theory Deep Dive

Time as domain data, not just formatting

Mental Model

Date and time bugs happen when code confuses human calendar concepts with machine timestamps. These are fundamentally different things:

  • A birthday is a LocalDate — it doesn't belong to a timezone. "January 1st" means the same thing in Tokyo or London.
  • A server log entry is an Instant — a nanosecond-precise UTC moment. Time zone is irrelevant for ordering events.
  • A calendar appointment in Mumbai at 2 PM is a ZonedDateTime — it anchors a wall-clock time to a specific political timezone that may observe DST.

Time zones are political constructs, not math. The IANA timezone database is updated when governments change DST rules. Offsets change. Storing +05:30 is different from storing Asia/Kolkata — the latter carries DST history. Always use the IANA zone ID for recurring schedules.

Duration and Period exist for a reason: one month is NOT 30 days. One hour on the day clocks spring forward is 60 minutes, but on fall-back day there are two 1 AM hours. The API forces you to be explicit.

Beginner-Friendly, Expert-Useful Lens

If you are new: Beginners should learn LocalDate, LocalTime, LocalDateTime, Instant, Duration, Period, and ZoneId separately.

If you are revising: Experts should refresh DST gaps, offset versus zone, clock injection, parsing formats, persistence, and testability.

Mastery check: You understand Date and Time when you can explain the simple mental model, write a tiny runnable example, identify the common trap, and describe how it behaves in a real project.

Why Legacy Date/Calendar Was a Disaster

Mutable
Not Thread-Safe
Date fields can be changed after creation, causing race conditions.
0-Indexed Months
January = 0
Calendar.JANUARY == 0, Calendar.DECEMBER == 11. Off-by-one bugs everywhere.
Poor API
Year = year - 1900
new Date(2024, 0, 1) means Jan 1 2024? No! It's year 3924.
Timezone Bugs

Core java.time Type Hierarchy

LocalDate
Date Only
No time, no zone. Use for birthdays, holidays, expiry dates.
LocalTime
Time Only
No date, no zone. Use for business hours, recurring daily alarms.
LocalDateTime
Date + Time
No timezone. Use for schedule display in known-context apps.
Instant
UTC Timestamp
Machine time. Best for logs, DB storage, event ordering.
ZonedDateTime
Full Precision
Date + Time + Zone rules. Use for user-facing meeting times.
OffsetDateTime
Fixed Offset
No DST rules, just an offset. Used in REST APIs and JSON.
JavaCreating instances
import java.time.*;
import java.time.temporal.*;

public class Main {
    public static void main(String[] args) {
        LocalDate today      = LocalDate.now();               // 2026-05-17
        LocalDate birthday   = LocalDate.of(1990, Month.JUNE, 15);
        LocalTime noon       = LocalTime.of(12, 0);
        LocalDateTime ldt    = LocalDateTime.of(today, noon);
        Instant now          = Instant.now();                 // UTC nanoseconds
        ZonedDateTime zdt    = ZonedDateTime.now(ZoneId.of("Asia/Kolkata"));
        OffsetDateTime odt   = OffsetDateTime.now(ZoneOffset.ofHoursMinutes(5, 30));

        System.out.println("Today: " + today);
        System.out.println("Birthday: " + birthday);
        System.out.println("LocalDateTime: " + ldt);
        System.out.println("Instant: " + now);
        System.out.println("ZonedDateTime: " + zdt);
        System.out.println("OffsetDateTime: " + odt);
    }
}

Date Arithmetic: Plus, Minus, Until

JavaImmutable arithmetic — always capture return value!
import java.time.*;
import java.time.temporal.*;

public class Main {
    public static void main(String[] args) {
        LocalDate today = LocalDate.now();
        LocalDate birthday = LocalDate.of(1990, 6, 15);

        LocalDate nextWeek    = today.plusWeeks(1);
        LocalDate lastMonth   = today.minusMonths(1);
        LocalDate anniversary = today.withDayOfMonth(1).withMonth(1);  // Jan 1 this year

        long daysBetween = ChronoUnit.DAYS.between(today, nextWeek);  // 7
        long monthsBetween = ChronoUnit.MONTHS.between(birthday, today);

        // Duration: machine time (seconds/nanos)
        Duration twoHours = Duration.ofHours(2);
        Instant later = Instant.now().plus(twoHours);

        // Period: human calendar (days/months/years)
        Period threeMonths = Period.ofMonths(3);
        LocalDate futureDate = today.plus(threeMonths);

        System.out.println("Next week: " + nextWeek);
        System.out.println("Days between: " + daysBetween);
        System.out.println("Months since birthday: " + monthsBetween);
        System.out.println("3 months from now: " + futureDate);
    }
}
WARN
Duration vs Period: Duration.ofDays(30) is always 2,592,000 seconds. Period.ofMonths(1) is calendar-aware — February + 1 month = March, not 28 days later.

Formatting and Parsing

JavaDateTimeFormatter — always thread-safe when created as constant
import java.time.*;
import java.time.format.*;
import java.util.*;

public class Main {
    public static void main(String[] args) {
        // Built-in ISO formats (preferred for APIs)
        String iso = LocalDate.now().format(DateTimeFormatter.ISO_LOCAL_DATE); // "2026-05-17"
        Instant parsed = Instant.parse("2026-05-17T10:30:00Z");

        // Custom pattern
        DateTimeFormatter fmt = DateTimeFormatter.ofPattern("dd MMM yyyy, hh:mm a", Locale.ENGLISH);
        String display = LocalDateTime.now().format(fmt); // "17 May 2026, 02:30 PM"

        // Parsing user input safely
        DateTimeFormatter userFmt = DateTimeFormatter.ofPattern("d/M/yyyy");
        LocalDate userDate = LocalDate.parse("5/3/2026", userFmt); // March 5

        // Locale-aware formatting
        DateTimeFormatter localeFmt = DateTimeFormatter
            .ofLocalizedDate(FormatStyle.LONG)
            .withLocale(Locale.forLanguageTag("hi-IN")); // Hindi format

        System.out.println("ISO date: " + iso);
        System.out.println("Custom format: " + display);
        System.out.println("Parsed user date: " + userDate);
        System.out.println("Hindi format: " + userDate.format(localeFmt));
    }
}
TIP
Declare DateTimeFormatter as a static final constant. It is immutable and thread-safe, unlike the legacy SimpleDateFormat which causes subtle concurrent bugs.

Business Date Math: TemporalAdjusters

Beyond simple addition, TemporalAdjusters handle real-world business date logic:

Java
import java.time.*;
import java.time.temporal.*;

public class Main {
    public static void main(String[] args) {
        LocalDate today = LocalDate.now();

        // Built-in adjusters
        LocalDate firstOfMonth  = today.with(TemporalAdjusters.firstDayOfMonth());
        LocalDate lastOfMonth   = today.with(TemporalAdjusters.lastDayOfMonth());
        LocalDate firstOfNext   = today.with(TemporalAdjusters.firstDayOfNextMonth());
        LocalDate nextFriday    = today.with(TemporalAdjusters.next(DayOfWeek.FRIDAY));
        LocalDate prevMonday    = today.with(TemporalAdjusters.previousOrSame(DayOfWeek.MONDAY));

        // Custom adjuster: next working day (skip weekends)
        TemporalAdjuster nextWorkingDay = TemporalAdjusters.ofDateAdjuster(date -> {
            DayOfWeek dow = date.getDayOfWeek();
            int daysToAdd = switch (dow) {
                case FRIDAY   -> 3;
                case SATURDAY -> 2;
                default       -> 1;
            };
            return date.plusDays(daysToAdd);
        });
        LocalDate nextWorkDay = today.with(nextWorkingDay);

        // Payment due date: last business day of month
        LocalDate endOfMonth = today.with(TemporalAdjusters.lastDayOfMonth());
        while (endOfMonth.getDayOfWeek() == DayOfWeek.SATURDAY
            || endOfMonth.getDayOfWeek() == DayOfWeek.SUNDAY) {
            endOfMonth = endOfMonth.minusDays(1);
        }

        System.out.println("Next Friday: " + nextFriday);
        System.out.println("Next work day: " + nextWorkDay);
        System.out.println("Last business day of month: " + endOfMonth);
    }
}

Timezone-Safe Design Patterns

JavaUTC storage, local display
import java.time.*;
import java.util.*;

public class Main {
    public static void main(String[] args) {
        // ✅ CORRECT PATTERN: Store in UTC, display in user's zone
        Instant eventTime = Instant.now();             // Store this in DB
        ZoneId userZone = ZoneId.of("America/New_York");
        ZonedDateTime display = eventTime.atZone(userZone);
        System.out.println("NY time: " + display);

        // ✅ Convert between zones correctly
        ZonedDateTime kolkata = ZonedDateTime.now(ZoneId.of("Asia/Kolkata"));
        ZonedDateTime london  = kolkata.withZoneSameInstant(ZoneId.of("Europe/London"));
        System.out.println("Kolkata: " + kolkata);
        System.out.println("London:  " + london);

        // ❌ WRONG: LocalDateTime loses timezone context
        LocalDateTime wrong = LocalDateTime.now(); // Whose time zone? Unknown!

        // ✅ Querying available zone IDs
        long zoneCount = ZoneId.getAvailableZoneIds().size(); // ~600 zones
        System.out.println("Available zones: " + zoneCount);

        // ✅ ZonedDateTime comparison is instant-based
        ZonedDateTime meeting1 = ZonedDateTime.parse("2026-06-01T09:00+05:30[Asia/Kolkata]");
        ZonedDateTime meeting2 = ZonedDateTime.parse("2026-06-01T04:00Z");
        boolean sameMoment = meeting1.toInstant().equals(meeting2.toInstant()); // true!
        System.out.println("Same instant? " + sameMoment);
    }
}
WARN
Daylight Saving Time trap: ZonedDateTime.of(2026, 3, 8, 2, 30, 0, 0, ZoneId.of("America/New_York")) will silently adjust the time because 2:30 AM doesn't exist on DST spring-forward day. Test schedule logic around March and November in US zones.

Legacy Interoperability

When integrating with older frameworks (JDBC, Hibernate, older libs) you'll need bridge conversions:

JavaBridge between java.util and java.time
import java.time.*;
import java.sql.Timestamp;
import java.util.*;

public class Main {
    public static void main(String[] args) {
        // java.util.Date ↔ Instant
        Date legacy  = new Date();
        Instant inst = legacy.toInstant();
        Date back    = Date.from(inst);

        // java.util.Calendar ↔ ZonedDateTime
        Calendar cal = Calendar.getInstance();
        ZonedDateTime zdt = cal.toInstant().atZone(cal.getTimeZone().toZoneId());

        // java.sql.Date ↔ LocalDate (for JDBC)
        java.sql.Date sqlDate  = java.sql.Date.valueOf(LocalDate.now());
        LocalDate localDate    = sqlDate.toLocalDate();

        // java.sql.Timestamp ↔ LocalDateTime / Instant
        Timestamp ts           = Timestamp.from(Instant.now());
        LocalDateTime fromTs   = ts.toLocalDateTime();

        // For modern JDBC drivers (PostgreSQL 42.x+, MySQL 8.x+)
        // directly pass LocalDate, LocalDateTime, Instant in PreparedStatement
        // ps.setObject(1, LocalDate.now());     // works with JDBC 4.2+
        // ps.setObject(2, Instant.now());       // maps to TIMESTAMP WITH TIME ZONE

        System.out.println("Legacy Date: " + legacy);
        System.out.println("LocalDate: " + localDate);
        System.out.println("From Timestamp: " + fromTs);
    }
}
INFO
Modern JDBC drivers support setObject() with java.time types directly. Prefer this over manual conversion to Timestamp.

Duration vs Period: Quick Reference

Duration
Machine Time
Hours, minutes, seconds, nanos. Fixed amount. Works with Instant and LocalTime. Duration.ofHours(2)
Period
Calendar Time
Years, months, days. Variable in seconds. Works with LocalDate. Period.ofMonths(3)
ChronoUnit
Unit Counting
Count units between two temporals. ChronoUnit.DAYS.between(a, b)

Principles That Matter

  • Always use java.time (Java 8+). Never use java.util.Date or Calendar in new code.
  • All java.time classes are immutable and thread-safe. You must capture return values — date.plusDays(1) returns a new object.
  • Store timestamps in UTC (Instant). Convert to local time only at display boundaries.
  • Use ZoneId by name (e.g., "Asia/Kolkata"), not by offset ("+05:30"), for DST safety.

Common Misunderstandings

  • ⚠ Using Date/Calendar in new code — They are flawed, mutable, and legacy. No exceptions.
  • ⚠ Forgetting to reassign — date.plusDays(1) does NOT modify date. You must write date = date.plusDays(1).
  • ⚠ LocalDateTime is not timezone-aware — It is a wall clock on an unknown wall. Never store it as a global timestamp.
  • ⚠ Serializing LocalDateTime to databases — Store as TIMESTAMP WITH TIME ZONE or map to Instant.

Hard-Won Tips

  • ⚠ LocalDateTime is not a moment on the global timeline. Without a zone or offset, it cannot identify an instant.
  • Daylight saving time creates missing and repeated local times. Scheduling systems must decide what to do in both cases.
  • Inject Clock in code that depends on now. It turns time from an untestable global into a controllable dependency.
  • Store instants for event timestamps, but preserve user zone when the original civil meaning matters.
  • Never parse dates with casual string splitting when DateTimeFormatter can encode format, locale, and validation.

Mastery Checklist

  • You can choose the right type: LocalDate, LocalDateTime, ZonedDateTime, or Instant.
  • You know the difference between Duration and Period.
  • You can format, parse, and convert between legacy and modern types.
  • You can use TemporalAdjusters for business date math.
  • You can explain why DST makes certain duration arithmetic incorrect.

Interview Preparation

Question 1
Why was java.time introduced?
Legacy Date/Calendar were mutable (not thread-safe), had terrible API design (0-indexed months, year = year-1900), mixed human and machine time concepts, and had poor timezone handling. java.time is immutable, clearly separates concepts, and has a fluent, consistent API.
Question 2
LocalDateTime vs ZonedDateTime vs Instant?
LocalDateTime = wall clock with no location (no zone). ZonedDateTime = wall clock with explicit timezone + DST rules. Instant = UTC nanoseconds since epoch, no zone concept. For global event storage, always use Instant.
Question 3
Duration vs Period?
Duration measures time in fixed units (seconds, nanos) — "2 hours = 7200 seconds always". Period measures in calendar units (years, months, days) — "1 month" varies by calendar month length and is not a fixed number of seconds.
Question 4
Why is SimpleDateFormat not thread-safe?
It stores parsing state in instance fields, so concurrent access corrupts results. DateTimeFormatter is immutable and always thread-safe — declare it static final and share freely.
