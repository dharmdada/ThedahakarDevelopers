Date and Time API

Use modern, immutable date/time types to avoid timezone bugs and fragile legacy code.

TH

Theory Deep Dive

Time as domain data, not just formatting

Mental Model

Date and time bugs happen when code confuses human calendar concepts with machine timestamps. A birthday is a LocalDate. A log entry is an Instant. A meeting scheduled for a city is a ZonedDateTime. Choosing the right type prevents entire classes of bugs.

Time zones are political and historical rules, not simple offsets. Offsets can change because of daylight saving or government decisions. Store global events as Instants and convert to local representation at the boundary where a user views them.

Duration and Period are different because humans think in calendar units while machines measure elapsed time. One month is not a fixed number of seconds; one hour during a DST transition may not behave like ordinary elapsed time.

Formatting and Parsing

Java
DateTimeFormatter f = DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss");
LocalDateTime now = LocalDateTime.now();
String text = now.format(f);

LocalDate parsed = LocalDate.parse("2026-05-16");
Instant instant = Instant.parse("2026-05-16T12:00:00Z");

Timezone-Safe Design

  • Store event instants in UTC (Instant).
  • Convert to user timezone only at display boundaries.
  • Never assume local timezone in backend business logic.
  • Use ZoneId explicitly for recurring schedules.
WARN
Daylight Saving Time changes can break assumptions. Test schedule logic around DST transition dates.

Legacy Interop

Java
// java.util.Date -> Instant
Date legacy = new Date();
Instant modern = legacy.toInstant();

// Instant -> java.util.Date
Date backToLegacy = Date.from(modern);

Prefer java.time in new code. Keep legacy conversions at integration boundaries only.
